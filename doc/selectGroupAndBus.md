# Техническое задание на разработку модуля selectGroupAndBus и внедрение его в существующий проект

## Описание бизнесс логики приложения
По бизнес-логике есть группы детей. У каждой группы есть свой Betreuer (пользователь с правами "user"). У каждой группы их может быть несколько. У каждого Betreuer обычно только одна группа. Привязка бетроер-группа осуществляется каждый день. Может не меняться, то есть все время один бетроер ведет одну группу, но могут и меняться группы. Betreuer может прийти, может не прийти, поэтому динамически привязывается группа к Betreuer через таблицу User_group_day

Пользователь входит в систему с помощью LoginView.vue. вход в систему нужно выполнять только один раз при регистрации, дальше эти данные сохраняются и пользователь входит в систему автомватически без ввода login+password (нет необходимости каждый раз их вводить) То есть LoginView.vue он видит всего один раз при регистрации, и это нормально.

Но каждый день он должен привязываться к группе и подтверждать свое участие в мероприятии. то есть перед тем как что-либо делать в системе, он должен нажимать на некую кнопочку например "Ich fahre heute mit" и тем самым подтверждать свое участие в поездке. При этом он должен обязательно указать с какой группой он едет (к какой группе он привязан) и в каком автобусе он будет ехать. то есть его рабочий день начинается с добавления записи в табличку user_group_day. туда он должен добавить свой user_ID, group_ID, bus_ID и установить status=1 , администратор может его добавить, но со статусом 0 (status = 0), поскольку он не знает будет ли user ехать или нет, но статус единичка означает, что user  в этот день физически  присутствует на мероприятии

## структура таблицы public.user_group_day
create table public.user_group_day (

id bigint generated by default as identity not null,

created_at timestamp with time zone not null default now(),

day date null,

user_id bigint null,

group_id smallint null,

status smallint null default '0'::smallint,

description character varying null,

bus_id smallint null,

constraint user_group_day_pkey primary key (id),

constraint user_group_day_user_id_fkey foreign KEY (user_id) references users (id) on update CASCADE on delete CASCADE

) TABLESPACE pg_default;

## Возможный Вариант технической реализации модуля
Предлагаю архитектурное решение, основанное на Глобальном Модальном Окне (Global Intercept Modal). Это самый надежный способ гарантировать выполнение ежедневного требования до того, как пользователь получит доступ к основным функциям приложения.

Предлагаемое решение: Модальное окно "Ich fahre heute mit"

## предложение по внедрению
1. Логика отображения (Интерсепт)
В  главном компоненте Vue (например, App.vue или в главном Layout) должен быть установлен глобальный Listener (наблюдатель), который срабатывает после успешной аутентификации.

Проверка Аутентификации: 
Вход пользователя в систему оставляем без изменения. Все должно быть так, как это реализовано сейчас:
Приложение загружается и автоматически проверяет наличие сохраненной сессии и сохраненых в localstorage (email + password) Если сессия есть или удается залогиниться "in silent mode" берем user_id и переходим к ежедневной проверке (new) Если нет — показывается LoginView.vue 

Ежедневная Проверка: Как только userId определен, приложение выполняет запрос к таблице user_group_day для текущего дня (day = today).

Если запись не найдена ИЛИ найдена, но status=0 (добавлено администратором): Устанавливается глобальное состояние isCheckInRequired = true.

Результат: Показывается полноэкранное модальное окно  selectGroupAndBus , которое нельзя закрыть без отправки формы. 

2. Структура формы (Модальное окно selectGroupAndBus)
Модальное окно должно быть максимально простым и сфокусированным на двух ключевых действиях.

Элемент UI	Назначение
Заголовок:	{{current date}} 
Dropdown 1: Gruppe: <select> с выбором из списка доступных групп [1, 2, ... useConfig.value.total_groups]
Dropdown 2:	Bus: <select> с выбором из списка доступных автобусов (Bus ID) [1,2, ... useConfig.value.total_buses]
Кнопка действия:	"Ich fahre heute mit!"


Дополнительный сценарий: Если администратор уже создал запись с status=0, форма должна быть предварительно заполнена данными, но кнопка по-прежнему требует нажатия для установки status=1 (подтверждение).

3. Возможность изменения данных в течение дня
Чтобы пользователь мог изменить группу/автобус (Requirement 5), необходимо обеспечить доступ к этой же форме, но уже в виде обычной, закрываемой модалки.

Где разместить: В основном меню (Sidebar) или в шапке (Header) приложения должен быть постоянные индикаторы, например:

 👥 Group_2 🚌 Bus_1 

При нажатии на эти индикаторы открываются  модальные окна selectGroup  и selectBus соответственно. Должна быть возможность изменить их независимо друг от друга (modal window "selectGroup", modal window "selectBus")



## Дополнительная информация по базе данных
Как таковой таблицы "groups" не существует. Там просто нечего хранить. Группы не имеют названия, только порядковый номер: 1,2, ... , config.total_groups
То же самое касается автобусов. Они имеют только порядковый номер: 1,2,...,config.total_buses.
