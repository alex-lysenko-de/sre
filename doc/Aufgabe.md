# Техническое задание: Система управления пользователями и ежедневной регистрацией

## 🎯 Цель проекта

Разработать комплексное решение для управления пользователями и их ежедневной регистрацией в системе, включающее:

1. **Модуль `useUser.js`** — композабл для работы с данными пользователя, расписанием и кэшированием
2. **Модуль `selectGroupAndBus`** — система модальных окон для ежедневной регистрации и привязки к группе/автобусу
3. **Интеграция** обоих модулей в существующий проект с автоматическим интерсептом

---

## 📋 Бизнес-логика приложения

### Общее описание

Приложение управляет работой Betreuer (воспитателей), которые сопровождают группы детей. Ключевые особенности:

- **Группы**: имеют только порядковый номер (1, 2, ..., `config.total_groups`), без названий
- **Автобусы**: также нумеруются порядково (1, 2, ..., `config.total_buses`)
- **Betreuer**: пользователи с ролью `'user'`, обычно закреплены за одной группой
- **Динамическая привязка**: каждый день Betreuer может быть привязан к разной группе через таблицу `user_group_day`
- **Присутствие**: Betreuer может прийти или не прийти, поэтому требуется ежедневное подтверждение

### Процесс аутентификации

1. **Первый вход**: пользователь видит `LoginView.vue` только один раз при регистрации
2. **Последующие входы**: данные (email + password) сохраняются в `localStorage`, вход происходит автоматически
3. **Ежедневная регистрация**: каждый день пользователь должен подтвердить участие и привязаться к группе/автобусу

### Статусы присутствия

- **`bMustWorkToday = 1`**: администратор создал запись, но пользователь не подтвердил присутствие
- **`isPresentToday = 1`**: пользователь физически присутствует на мероприятии (подтвердил участие)

| Поле          | Тип                       | Описание                                     |
| ------------- | ------------------------- | -------------------------------------------- |
| `id`          | `bigint`                  | первичный ключ                               |
| `created_at`  | `timestamp with time zone`| дата/время создания записи                   |
| `day`         | `date`                    | дата мероприятия                             |
| `user_id`     | `bigint`                  | FK → `users.id` (CASCADE)                    |
| `group_id`    | `smallint`                | номер группы (1..total_groups)               |

- **Admin**: может создавать/редактировать записи для любого пользователя
- **User**: видит только свои записи, может создавать/обновлять только свои данные
- **Использование**: данные используются для импорта событий в календарь (например, Google Calendar)

## 🧩 Часть 1: Модуль `useUser.js`

### Назначение

Vue композабл для централизованного управления данными пользователя с кэшированием и методами работы с расписанием.

### Структура данных

```js
const userInfo = ref({
  // Из таблицы users
  id: null,              // users.id (bigint)
  user_id: null,         // users.user_id (uuid из auth)
  email: null,           // users.email
  display_name: null,    // users.display_name
  role: null,            // users.role ('admin' | 'user')
  
  // Из таблицы user_group_day (на сегодня)
  group_id: null,        // номер группы
  bus_id: null,          // номер автобуса
  bMustWorkToday: false, // true если есть запись на сегодня и bMustWorkToday==1
  isPresentToday: false  // true если запись на сегодня и isPresentToday==1
})
```

### Логика инициализации

1. **Получение auth пользователя**
   - Вызов `supabase.auth.getUser()`  // Это не точно
   - Извлечение `user.id` (uuid)

2. **Загрузка данных из `users`**
   - SELECT по `user_id = auth_uuid`
   - Заполнение базовых полей (`id`, `email`, `display_name`, `role`)

3. **Загрузка сегодняшнего расписания**
   - SELECT из `user_group_day` WHERE `user_id = users.id` AND `day = CURRENT_DATE`
   - Если запись найдена:
     - `bMustWorkToday`,`isPresentToday`, `group_id`, `bus_id`, `status` → в `userInfo`
   - Если не найдена:
     - `bMustWorkToday = false`
     - `isPresentToday = false`

### Кэширование

- **Ключ**: `user_info_cache`
- **TTL**: 10 минут
- **Структура кэша**:
  ```js
  {
    timestamp: Date.now(),
    data: { ...userInfo }
  }
  ```
- **Логика использования**: 
  - При вызове `loadUser(force = false)` проверяется актуальность кэша
  - Если `force = true` или кэш устарел — загрузка из Supabase
  - Иначе — восстановление из `localStorage`

### Публичный API

```js
export function useUser() {
  return {
    // Reactive состояние
    userInfo,          // ref({ id, user_id, email, ... })
    isAdmin,           // computed(() => userInfo.value.role === 'admin')
    loading,           // ref(boolean)
    error,             // ref(Error | null)
    
    // Методы загрузки
    loadUser,          // async (force = false) — с кэшем
    fetchUserFromSupabase, // async () — принудительно из БД
    
    // Методы работы с user_group_day
    assignUserToGroup,   // async (groupId, date = today)
    assignUserToBus,     // async (busId, date = today)
    updateUserPresence,  // async (status, date = today)
    getUserDayInfo,      // async (date = today) → { group_id, bus_id, status }
    
    // Утилиты
    clearUserCache     // () => void — очистка при logout
  }
}
```

### Описание методов

#### `loadUser(force = false)`
- Проверяет кэш, загружает данные если нужно
- Объединяет данные из `users` и `user_group_day`
- Сохраняет результат в кэш

#### `assignUserToGroup(groupId, date = today)`
- Создает новую запись в `user_group_day` если её нет
- Обновляет `group_id` если запись существует
- UPDATE `userInfo.value.group_id`

#### `assignUserToBus(busId, date = today)`
- Аналогично `assignUserToGroup`, но для `bus_id`

#### `updateUserPresence(bPresense, date = today)`
- UPDATE `user_group_day` SET `isPresentToday` = 0 или 1
- Обновляет `isPresentToday` в `userInfo`

#### `getUserDayInfo(date)`
- SELECT расписания на конкретную дату
- Возвращает объект `{ group_id, bus_id, bMustWorkToday, isPresentToday, description }`

### Алгоритм работы при загрузке приложения

```
1. App загружается
   │
2. Проверка auth сессии (supabase.auth.getSession())
   │
   ├─→ Сессии НЕТ
   │   ├─→ Проверка localStorage (email + password)
   │   │   ├─→ Найдено → Silent login (supabase.auth.signInWithPassword)
   │   │   │   ├─→ Успех → Переход к шагу 3
   │   │   │   └─→ Ошибка → Показать LoginView.vue
   │   │   └─→ НЕ найдено → Показать InfoView.vue
   │   │
   │   └─→ Показать InfoView.vue
   │ 
   │
   └─→ Сессия ЕСТЬ
       │
3. Загрузка useUser.loadUser()
   │
4. Проверка ежедневной регистрации 
   │
   ├─→ bMustWorkToday = false (нет записи на сегодня)
   │   └─→ isCheckInRequired = true
   │
   ├─→ bMustWorkToday = true, но isPresentToday = false (status = 0)
   │   └─→ isCheckInRequired = true
   │
   └─→ bMustWorkToday = true И isPresentToday = true (status = 1)
       └─→ isCheckInRequired = false
       
5. Если isCheckInRequired = true
   │
   └─→ Показать НЕОТКЛЮЧАЕМУЮ модалку "Ich fahre heute mit"
       │
       ├─→ Пользователь заполняет форму (group_id, bus_id)
       ├─→ Нажимает "Ich fahre heute mit!"
       ├─→ Вызов assignUserToGroup(group_id)
       ├─→ Вызов assignUserToBus(bus_id)
       ├─→ Вызов updateUserPresence(1)
       └─→ isCheckInRequired = false, модалка закрывается
       
6. Доступ к основному функционалу приложения
```

### Логика компонента

**Входные параметры:**
- Диапазоны из `useConfig`: `total_groups`, `total_buses`
- Текущие данные из `useUser`: `userInfo.value.group_id`, `userInfo.value.bus_id`

**Предзаполнение формы:**
- Если администратор уже создал запись (bMustWorkToday: 1), поля предзаполняются
- Кнопка остается активной — требуется подтверждение

**Валидация:**
- Оба поля обязательны для заполнения
- Кнопка неактивна пока оба dropdown не выбраны

**Обработка отправки:**
```js
async function onSubmit() {
  loading.value = true
  try {
    await assignUserToGroup(selectedGroup.value)
    await assignUserToBus(selectedBus.value)
    await updateUserPresence(1)
    isCheckInRequired.value = false // Закрыть модалку
  } catch (err) {
    // Показать ошибку, НЕ закрывать модалку
    error.value = err.message
  } finally {
    loading.value = false
  }
}
```

**Особенности:**
- Модалка НЕ имеет кнопки закрытия (крестика)
- НЕ закрывается по клику вне модалки
- НЕ закрывается по ESC
- Блокирует доступ ко всему остальному UI, (кроме кнопки "Abmelden")

## 📝 Сценарии использования

### Сценарий 1: Новый рабочий день (нет записи)

1. Пользователь открывает приложение
2. Автоматический вход (сохраненная сессия)
3. `useUser.loadUser()` определяет `bMustWorkToday = false`
4. Показывается блокирующая модалка "Ich fahre heute mit"
5. Пользователь выбирает Gruppe 2, Bus 1
6. Нажимает кнопку
7. Создается запись: `{ day: today, user_id, group_id: 2, bus_id: 1,  bMustWorkToday: 0, isPresentToday = 1  }`
8. Модалка закрывается, доступ к приложению открыт

### Сценарий 2: Администратор создал запись заранее

1. Админ добавил запись: `{ day: today, user_id, group_id: 3, bus_id: 2, bMustWorkToday: 1, isPresentToday = 0 }`
2. Пользователь открывает приложение
3. `useUser.loadUser()` определяет `bMustWorkToday = true`, но `isPresentToday = false`
4. Показывается модалка с предзаполненными полями (Gruppe 3, Bus 2)
5. Пользователь может изменить значения или оставить как есть
6. Нажимает "Ich fahre heute mit!"
7. UPDATE `isPresentToday = 1`
8. Модалка закрывается

### Сценарий 3: Изменение группы в течение дня

1. Пользователь уже зарегистрирован (status = 1)
2. В Header видит индикатор "👥 Gruppe 2"
3. Кликает на индикатор
4. Открывается модалка "Gruppe ändern"
5. Выбирает Gruppe 4
6. Нажимает "Speichern"
7. UPDATE `group_id = 4` в текущей записи
8. Индикатор обновляется: "👥 Gruppe 4"

### Сценарий 4: Администратор просматривает данные

1. Админ входит в систему
2. `isAdmin.value = true`
3. Видит расширенный интерфейс с таблицей всех пользователей
4. Может создавать/редактировать записи в `user_group_day` для любого пользователя
5. Может устанавливать `status = 0` (запланировано) или `1` (подтверждено)

## 🔐 Безопасность и права доступа

### Row Level Security (RLS) в Supabase

**Таблица `users`:**
-- права доступа уже установлены

**Таблица `user_group_day`:**
```sql
-- Пользователь видит только свои записи
CREATE POLICY "Users view own schedule"
  ON user_group_day FOR SELECT
  USING (
    user_id = (SELECT id FROM users WHERE user_id = auth.uid())
  );

-- Пользователь может изменять только свои записи
CREATE POLICY "Users update own schedule"
  ON user_group_day FOR UPDATE
  USING (
    user_id = (SELECT id FROM users WHERE user_id = auth.uid())
  );

-- Админы имеют полный доступ
CREATE POLICY "Admins full access"
  ON user_group_day FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM users
      WHERE user_id = auth.uid() AND role = 'admin'
    )
  );
```

## 🧪 Тестовые сценарии

### Модуль useUser.js

1. **Кэширование**:
   - Первая загрузка → запрос в БД
   - Повторная загрузка в течение 10 мин → из кэша
   - Загрузка через 11 мин → новый запрос в БД
   - `loadUser(force: true)` → игнорирует кэш

2. **Создание записи**:
   - Нет записи на сегодня → `assignUserToGroup` создает новую
   - Есть запись → `assignUserToGroup` обновляет существующую

3. **Обновление статуса**:
   - `updateUserPresence(1)` → `isPresentToday = true`
   - `updateUserPresence(0)` → `isPresentToday = false`

### Модальные окна

1. **Блокирующая модалка**:
   - Открывается если `bMustWorkToday = false`
   - Открывается если `isPresentToday = false`
   - НЕ закрывается по ESC, по клику вне, по крестику
   - Закрывается только после успешной отправки формы

2. **Изменяемые модалки**:
   - Открываются по клику на индикаторы
   - Закрываются по ESC, по клику вне, по крестику
   - Сохранение → UPDATE в БД + закрытие модалки

3. **Предзаполнение**:
   - Если admin создал запись на текущий день → поля предзаполнены
   - Если записи нет → поля заполнены из последнего дня 
   - Если записи в таблице user_group_day отсутствуют вообще → поля пустые
