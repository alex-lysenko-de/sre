# IMPLEMENTATION_PLAN — Ticket 102

## Тикет
На странице `https://sre-nrw.de/days-edit` при нажатии на кнопку "корзина" (удаление дня) запись пропадает из списка, но при обновлении страницы появляется снова — то есть удаление в БД фактически не происходит.

---

## Цель

Устранить ситуацию, при которой пользователь получает ложное подтверждение удаления дня (`Der Tag "..." wurde erfolgreich gelöscht.`), хотя запись физически остаётся в таблице `public.days`. Исправление должно работать на двух уровнях: (1) БД должна реально разрешать/запрещать операцию в соответствии с правами пользователя, (2) клиент должен корректно отличать "удаление прошло" от "удаление молча не выполнено" и не показывать успех там, где его не было.

---

## Диагноз (root cause)

`deleteDay()` в `src/composables/useDays.js`:

```js
const { error } = await supabase
    .from('days')
    .delete()
    .eq('id', dayId)

if (error) { ... }
return true
```

Это классическая для Supabase/PostgREST ловушка: если RLS-политика не разрешает текущему пользователю удалить строку (или для таблицы вовсе нет `DELETE`-политики при включённом RLS), PostgREST не считает это ошибкой — предикат RLS просто исключает строку из выборки для удаления, запрос выполняется "успешно" и удаляет **0 строк**. Supabase-клиент возвращает `error: null`, `data: null`. Функция `deleteDay()` в таком случае возвращает `true`, `removeDay()` в `DaysEditView.vue` (`src/views/DaysEditView.vue:371-390`) считает это успехом, убирает элемент из локального массива `this.days` и показывает "erfolgreich gelöscht". После перезагрузки страницы `fetchDaysList()` заново читает из БД — а строка там и оставалась всё это время.

Это ровно тот же класс проблемы, который уже был обнаружен и **частично** обработан для другой операции в этом же файле: в `saveDay()` для веток `UPDATE` есть явная защита именно от этого сценария —

```js
// ADDITIONAL CHECK FOR UPDATE:
if (id && (!data || data.length === 0)) {
    throw new Error(`RLS error: Update not performed...`)
}
```

и комментарии в коде прямо ссылаются на RLS-политики `days` ("Ensure the RLS policy for INSERT/UPDATE on table 'days' is set to..."). То есть автор уже знал и учитывал, что RLS для `days` активен и может тихо блокировать операции — но соответствующая защита для `DELETE` не была добавлена. Баг №102 — это тот же паттерн, не закрытый для одного из четырёх CRUD-путей.

Второй слой проблемы — **отсутствие документации RLS-политик для `days`**. Ни в `doc/database_migration.sql` (раздел 7, "Enable Row Level Security"), ни в `doc/project.md`, ни в `doc/neu 1.txt` для таблицы `days` не описано ни одной RLS-политики (в отличие от `c_bands`, `children`, `scans`, `groups`, для которых политики задокументированы явно). При этом код (`saveDay`) косвенно подтверждает, что RLS на `days` в реальной БД включён. Из этого следует практический вывод: политики на `days`, скорее всего, создавались вручную через Supabase Dashboard, без синхронизации с миграционными файлами репозитория — и, вероятно, туда просто забыли добавить политику для `DELETE`. Это тот же класс архитектурного разрыва "код и БД разошлись с документацией", что уже фиксировался в плане тикета 101 для таблицы `config` — стоит рассматривать как системную проблему процесса, а не разовую случайность.

## Затрагиваемые модули

- `src/composables/useDays.js` — `deleteDay()`, единственное место, где выполняется DELETE-запрос к `days`.
- `src/views/DaysEditView.vue` — `removeDay()`, обработчик кнопки "корзина"; логика вызова уже корректна (ждёт `await deleteDay()` перед локальным удалением из массива), менять не требуется — она автоматически заработает правильно, как только `deleteDay()` начнёт кидать ошибку в нужном случае.
- Supabase: таблица `public.days`, её RLS-политики (требуется аудит и, скорее всего, миграция).
- Документация: `doc/database_migration.sql` (или отдельный migration-файл) — должна быть приведена в соответствие с реальным состоянием RLS в проде, чтобы не повторять этот класс бага для других таблиц.

## Изменения в существующих модулях

### `src/composables/useDays.js` — `deleteDay()`
- Добавить `.select()` к DELETE-запросу, чтобы получить список фактически удалённых строк (как уже сделано в `saveDay()` для UPDATE):
  ```js
  supabase.from('days').delete().eq('id', dayId).select()
  ```
- После запроса проверить: если `error` есть — пробросить его как сейчас; если `error` нет, но `data` пустой массив (0 строк реально удалено) — выбросить явную ошибку, аналогичную по духу существующей в `saveDay()`, например: "RLS error: Löschen wurde nicht durchgeführt. Die Sicherheitsrichtlinie erlaubt das Entfernen dieses Eintrags nicht." Формулировку/язык сообщения нужно выдержать в стиле уже существующих сообщений в этом файле (немецкий текст в UI-слое, комментарии на английском — как сейчас).
- Только при реально ненулевом списке удалённых строк функция должна возвращать успех.

### `src/views/DaysEditView.vue`
- Изменений в логике не требуется. `removeDay()` уже оборачивает `deleteDay()` в `try/catch` и корректно показывает `showAlert(..., 'danger')` при ошибке (строки 386-389) — после фикса composable этот путь начнёт срабатывать так, как задумано изначально, вместо того чтобы всегда идти по "успешной" ветке.

## Изменения БД

Требуется **аудит перед изменением**, а не миграция вслепую:

1. Проверить в Supabase Dashboard текущие RLS-политики таблицы `public.days`:
   - включён ли RLS вообще (судя по поведению `saveDay()` — да);
   - какие политики существуют для `SELECT` / `INSERT` / `UPDATE`;
   - есть ли политика для `DELETE` — по симптомам тикета, её либо нет, либо она заведомо ложная (`USING (false)` или условие, не выполняющееся для текущих пользователей).
2. Добавить/скорректировать политику `DELETE` для `public.days`, по аналогии с уже существующим паттерном админ-политик в `doc/database_migration.sql` (например, для `c_bands`: "Allow admins to manage bands" — `FOR ALL ... USING (EXISTS (SELECT 1 FROM users WHERE users.user_id = auth.uid() AND users.role = 'admin' AND users.active = true))`). Для `days` логично завести аналогичную политику `FOR ALL` (либо отдельно `FOR DELETE`), ограниченную ролью `admin`, — это согласуется с тем, что маршрут `/days-edit` уже защищён на уровне роутера (`meta: { requiresAuth: true, requiresAdmin: true }`).
3. Зафиксировать итоговый набор политик `days` в SQL-миграции в репозитории (например, дополнить `doc/database_migration.sql` или завести отдельный `doc/db/days_rls.sql`), чтобы устранить расхождение между кодом/Dashboard и документацией — предотвращает повторение этого класса бага для других таблиц в будущем.

Схема таблицы `days` (структура столбцов, типы) не меняется.

## API изменения

Изменений в Supabase Edge Functions не требуется. Меняется только форма клиентского запроса на удаление (`.delete().eq(...)` → `.delete().eq(...).select()`) и трактовка его результата — без изменения публичного контракта `useDays()` (сигнатура `deleteDay(dayId)` остаётся прежней, меняется только то, при каких условиях промис теперь будет реджектиться).

2. Подготовить и применить SQL-миграцию, добавляющую/исправляющую `DELETE`-политику для `days` (роль `admin`, по аналогии с существующими политиками для `c_bands`/`groups`).
3. Задокументировать актуальные RLS-политики `days` в репозитории (см. "Изменения БД", п.3), чтобы устранить расхождение код/Dashboard/документация.
4. Внести правку в `deleteDay()` (`src/composables/useDays.js`): `.select()` + проверка на пустой результат + явная ошибка.
5. Ручная проверка на `/days-edit`:
   - Удаление админом существующего дня → запись реально пропадает из БД, подтверждается перезагрузкой страницы.

## Риски

- **RLS-политики `days` неизвестны из репозитория** — есть риск, что при добавлении `DELETE`-политики без предварительного аудита будет создана дублирующая/конфликтующая политика, либо наоборот — слишком широкая (например, разрешающая удаление любому `authenticated`, а не только `admin`), что откроет доступ к удалению дней для обычных пользователей. Порядок действий (аудит — шаг 1, до миграции — шаг 2) специально минимизирует этот риск.
- Если для `days` окажется, что RLS в принципе не включён (несмотря на косвенные признаки из `saveDay()`), причина бага будет другой (например, триггер или проблема на стороне PostgREST/кэша схемы) — в этом случае план потребует пересмотра после аудита; сам факт "0 rows affected без ошибки" при DELETE под RLS — наиболее вероятная, но не единственная теоретическая причина, поэтому аудит обязателен перед написанием миграции.
- Правка `deleteDay()` без соответствующей корректной политики на `DELETE` не решит проблему — просто заменит "тихий провал" на видимую ошибку у всех пользователей. Оба изменения (БД + клиент) необходимы вместе; частичное внедрение (только клиент) не является приемлемым фиксом.
- Тот же паттерн "нет `.select()` + нет проверки на 0 строк" стоит поискать и в других composables, работающих с DELETE (например, `useChildren.js`, `useGroups.js`, `useArmband.js`) — вне scope этого тикета, но рекомендуется завести отдельный тикет на аудит, чтобы не столкнуться с той же проблемой на других экранах.
