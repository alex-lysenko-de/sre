### `doc/db/days_rls.sql` (новый файл)
Миграция, документирующая и приводящая в порядок RLS-политики `public.days`:
- `ALTER TABLE public.days ENABLE ROW LEVEL SECURITY`.
- `SELECT` — `TO authenticated`: страница `/days-edit` и связанный функционал доступны только залогиненным пользователям (маршрут защищён `meta: { requiresAuth: true, requiresAdmin: true }` в router), публичного чтения `days` в коде не найдено (в отличие от `config`/`/info` из тикета 101).
- `INSERT/UPDATE/DELETE` (`FOR ALL`) — только для `users.role = 'admin' AND users.active = true`, по образцу уже существующих политик для `c_bands`/`groups` в `doc/database_migration.sql`, с явным `WITH CHECK` в дополнение к `USING`.
- Политики idempotent (`DROP POLICY IF EXISTS` перед каждым `CREATE POLICY`), безопасно для повторного прогона.

Как и в тикете 101, сам факт наличия файла миграции в репозитории ничего не защищает, пока он не применён в самой Supabase — это отдельная обязательная ручная проверка (см. ниже).

## Отклонения от плана

- **Аудит текущих RLS-политик `public.days` в Supabase Dashboard не выполнен** — нет доступа к dashboard/CLI с продовыми credentials в рамках этой сессии разработки. План явно требовал аудит **до** написания миграции (шаг 1 "Последовательности реализации"), чтобы не создать дублирующую/конфликтующую политику.
  - Вместо этого миграция `doc/db/days_rls.sql` написана как "целевое состояние" по коду и по уже задокументированному в репозитории паттерну (`c_bands`, `groups` в `doc/database_migration.sql`), с `DROP POLICY IF EXISTS` перед каждым `CREATE POLICY` — это должно сделать применение безопасным независимо от текущего фактического состояния (перезапишет любые существующие одноимённые политики, не создав дублей). Но если в Dashboard уже есть политики `days` с **другими названиями**, они не будут удалены этой миграцией и могут конфликтовать (например, слишком широкая политика на DELETE, разрешающая любому `authenticated`) — это нужно проверить вручную перед/после применения (см. "Ручные проверки").
  - Проверка репозитория подтвердила диагноз плана: в `doc/database_migration.sql` для `days` есть только `CREATE TABLE`/`CREATE INDEX`/`COMMENT` (строки 89-102), но нет ни `ENABLE ROW LEVEL SECURITY`, ни `CREATE POLICY` — RLS для этой таблицы в репозитории не задокументирован вообще.
- Аудит других composables с DELETE-запросами без `.select()` (`useChildren.js`, `useGroups.js`, `useArmband.js` — упомянуты в плане как риск, но явно вне scope этого тикета) не проводился — план прямо указывает завести под это отдельный тикет.

## Требуемые миграции

`doc/db/days_rls.sql` должна быть применена в Supabase (SQL Editor или CLI) **до** релиза этого изменения — без неё фикс на уровне клиента (`deleteDay()`) не решает проблему целиком: он либо продолжит "тихо не удалять" (если DELETE-политики для `days` по-прежнему нет — тогда 0 удалённых строк теперь корректно покажет ошибку вместо ложного успеха, что уже является улучшением), либо будет неверно блокировать легитимные admin-удаления (если RLS почему-то настроен иначе, чем предполагает миграция). Реальную проблему тикета ("днём удаляются, но появляются снова") закрывает именно применение миграции, а не только правка клиента — это прямо отмечено в разделе "Риски" плана.

## Ручные проверки

Перед мержем/релизом обязательно выполнить:

1. **Аудит текущих RLS-политик `days`** в Supabase Dashboard (или `select * from pg_policies where tablename = 'days';`) — убедиться, нет ли уже политик с другими именами, которые могут конфликтовать с `doc/db/days_rls.sql`.
2. **Применить `doc/db/days_rls.sql`** в Supabase, затем подтвердить через `pg_policies`, что появились политики `Allow authenticated users to read days` (SELECT) и `Allow admins to manage days` (ALL, только admin+active).
3. **Основной сценарий бага**: на `/days-edit` под тестовым admin-аккаунтом удалить существующий день → запись реально пропадает из БД, подтверждается перезагрузкой страницы (именно это было симптомом тикета 102).
4. **Негативный сценарий**: попытка удаления уже несуществующего/чужого `id` (например, двойной быстрый клик) → должна показываться ошибка `Fehler beim Löschen des Tages: ...`, а не ложное "erfolgreich gelöscht".
5. **Проверка отказа на уровне БД**: прямой вызов Supabase API на удаление `days` от имени не-admin пользователя (если такой пользователь может быть аутентифицирован) должен быть отклонён политикой, а не только UI/роутером.
6. **Регрессия**: `saveDay()` (создание/редактирование дня) и остальной функционал `/days-edit` не затронуты правкой — стоит подтвердить вручную, так как миграция меняет весь набор политик `days` (`FOR ALL`), а не только `DELETE`.
7. `npx vite build` выполнен локально в рамках этой сессии — сборка проходит без ошибок, синтаксических проблем в изменённом файле нет. Это не заменяет функциональную/browser-проверку.
