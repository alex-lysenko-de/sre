## Minor

### 1. Стилистическая непоследовательность текста ошибки внутри одного файла

Новое сообщение в `deleteDay()`:
```js
throw new Error(`RLS error: Löschen wurde nicht durchgeführt. Die Sicherheitsrichtlinie erlaubt das Entfernen dieses Eintrags nicht.`)
```
смешивает английский префикс "RLS error:" с немецким текстом. Соседний, уже существующий код в `saveDay()` в этом же файле использует для аналогичной ситуации полностью английский текст ("RLS error: Update not performed. The security policy does not allow you to modify record with ID..."). План просил выдержать стиль "как сейчас", но в файле фактически нет единого стиля (`saveDay` — английский, остальные console.log в файле — немецкий). Выбор разработчика (немецкий текст, т.к. он в итоге идёт в пользовательский alert через `showAlert`) функционально оправдан и корректен, но стоит унифицировать оба сообщения (`saveDay`/`deleteDay`) в отдельном тикете по техдолгу, а не оставлять расхождение внутри одного composable.

### 2. Тот же класс бага подтверждённо существует в `useChildren.js`

Проверено чтением кода: `useChildren.js:166-168` содержит точно такой же паттерн, что и был багом в этом тикете —
```js
.delete()
.eq('id', childId);
```
без `.select()` и без проверки на 0 удалённых строк. План (раздел "Риски") прямо указывал это как вероятный, но не проверенный риск и рекомендовал отдельный тикет на аудит; проверка подтверждает, что риск не гипотетический, а реальный — минимум один живой экземпляр той же уязвимости уже найден. Не блокирует этот тикет (вне scope), но стоит завести тикет с higher приоритетом, чем "может быть, стоит проверить" — конкретный файл и строка уже известны.

# Архитектурные замечания

- Исправление `deleteDay()` — точная копия уже одобренного паттерна из `saveDay()` (`.select()` + проверка `data.length === 0` + явный `throw`), обеспечивает согласованность стиля обработки RLS-ошибок внутри одного composable.
- `removeDay()` в `DaysEditView.vue` не менялся, как и предполагал план — `try/catch` вокруг `await deleteDay()` уже был реализован корректно и просто не мог сработать из-за отсутствия ошибки от `deleteDay()`; после фикса composable путь ошибки автоматически заработал так, как задумано изначально. Проверено чтением `DaysEditView.vue:371-390` — код не тронут, логика верна.
- `doc/db/days_rls.sql` по структуре идентична `doc/database_migration_config_rls.sql` из тикета 101 (тот же паттерн `EXISTS (SELECT 1 FROM public.users WHERE users.user_id = auth.uid() AND users.role = 'admin' AND users.active = true)`), что хорошо для консистентности кодовой базы. `SELECT ... TO authenticated USING (true)` обоснован: подтверждено грепом, что таблица `days` читается только из `useDays.js`, а он используется только в `DaysEditView.vue` (маршрут `/days-edit`, `requiresAuth + requiresAdmin`) и частично в `AdminBusView.vue` (тоже `requiresAuth + requiresAdmin`, но `fetchDaysList`/чтение `days` там не вызывается) — публичных/гостевых путей чтения `days` не существует, значит ограничение `TO authenticated` (не `public`) корректно и не сломает ничего.
- `FOR ALL` политика на INSERT/UPDATE/DELETE одной записью, ограниченная `role = 'admin' AND active = true`, соответствует уже используемому в проекте паттерну (`c_bands`, `groups`) и корректно согласуется с тем, что маршрут уже гейтится на уровне роутера — это defense-in-depth, а не дублирующая логика.
- Как и в тикете 101, реальная защита появится только после применения миграции в Supabase — сам факт наличия SQL-файла в репозитории не является исполнением исправления. Разработчик явно и честно зафиксировал это ограничение в отчёте (аналогично прошлому тикету), что хорошо для прозрачности процесса.

# Рекомендации

1. **Перед релизом обязательно применить `doc/db/days_rls.sql`** к боевому Supabase и подтвердить через `pg_policies`, что политики `Allow authenticated users to read days` и `Allow admins to manage days` активны — без этого шага сам баг тикета (тихое неудаление) не будет устранён по факту, а лишь начнёт корректно показывать ошибку вместо ложного успеха (что уже является улучшением, но не полным решением).
2. Перед применением миграции стоит один раз проверить `pg_policies` на предмет уже существующих политик `days` с другими именами (риск, честно отмеченный разработчиком) — `DROP POLICY IF EXISTS` в миграции удалит только политики с точно такими же именами, не тронув потенциальные "чужие" политики, которые могут остаться и конфликтовать.
3. Завести отдельный тикет на `useChildren.js:166-168` — тот же класс бага (DELETE без `.select()`/проверки строк), уже подтверждён чтением кода, а не только предположением из плана. Также стоит включить в аудит `useGroups.js`/`useArmband.js`, упомянутые в плане (в этой сессии не проверялись подробно).
4. Опционально: унифицировать язык/стиль сообщений RLS-ошибок между `saveDay()` (английский) и `deleteDay()` (немецкий) в рамках техдолга.
