# Миграция в сторону multi-tenant

В `doc/QR-codes.md` описана целевая архитектура публичных URL для QR-кодов на браслетах, рассчитанная на несколько независимых организаций (tenants) в рамках одного приложения:

```text
https://sre-nrw.de/{tenant}/{id}
```

где `tenant` — двухбуквенный идентификатор организации (изначально `id`, в будущем — любой другой), а `id` — номер браслета в пределах этого tenant. После аутентификации tenant хранится в сессии/JWT пользователя и не передаётся в внутренних URL приложения (`/edit/101`, `/history/101` и т.д.); при сканировании QR-кода tenant из URL сверяется с tenant из сессии.

Сейчас приложение полностью single-tenant: во всех таблицах БД (`children`, `scans`, `days`, `users`, `groups`, `config` и др.) отсутствует понятие организации, а публичный URL браслета — просто `https://sre-nrw.de?id={id}`.

## Что нужно сделать в рамках этого тикета

Спланировать (без немедленной реализации) миграцию к multi-tenant архитектуре, описанной в `doc/QR-codes.md`:

* **Схема БД** — добавление `tenant`/`org_id` в затрагиваемые таблицы, стратегия миграции существующих данных единственного текущего tenant, RLS-политики с учётом tenant.
* **Маршрутизация** — переход публичных QR-URL с `?id={id}` на `/{tenant}/{id}`, правило маршрутизации "первый сегмент из двух латинских букв → tenant, иначе → команда приложения" (`/info`, `/preview`, `/login`, `/admin`, `/api`, ...).
* **Аутентификация и сессия** — где и как tenant пользователя сохраняется после логина, проверка совпадения tenant при сканировании QR-кода, поведение при несовпадении (см. раздел "Different tenant" в `doc/QR-codes.md`).
* **Обратная совместимость** — что делать с уже выпущенными браслетами со старым форматом URL (`?id={id}`) на переходный период.
* **Влияние на существующий код** — какие composables, stores и views из `src/` затронуты (в первую очередь всё, что работает с `children`/`scans`/маршрутом сканирования браслета).

## Результат

Итог тикета — план/архитектурное решение (implementation plan) для последующей реализации, а не сама реализация. План должен опираться на базу знаний проекта (тикет 108), если она к этому моменту уже готова, и учитывать текущую структуру БД, задокументированную в `doc/table_structure.md` и `doc/db/`.
