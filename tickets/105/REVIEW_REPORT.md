# Найденные проблемы

## Critical

Нет.

## Major

### 1. Отсутствует обработка ошибок при отказе LocalForage в `router/index.js:150` и `App.vue:235`

```js
// router/index.js:150, внутри router.beforeEach(async (to, from, next) => {...})
const isRegistered = await getAuthItem('sre_user_registered') === 'true'
```
```js
// App.vue:235, внутри async function initializeApp() {...}, вызывается из onMounted БЕЗ try/catch
const isRegistered = await getAuthItem('sre_user_registered') === 'true';
```

Оба вызова находятся **до** ближайшего `try/catch` в своей функции (`try` в `router/index.js` оборачивает только ветку `to.meta.requiresAuth`, `try` в `App.vue` начинается двумя строками ниже). До миграции `localStorage.getItem()` был синхронным и практически никогда не бросал исключение. Теперь `getAuthItem()` идёт через `readThrough()` → `authLocalForage.getItem(key)` (IndexedDB), которая может отклонить промис — ровно в тех же условиях, ради устойчивости к которым и делается вся эта миграция (IndexedDB заблокирован/недоступен в приватном режиме Safari, quota exceeded, ITP-ограничения и т.п., см. риски в `IMPLEMENTATION_PLAN.md`).

**Сценарий отказа:** на iOS в состоянии, когда IndexedDB недоступен (и по каким-то причинам не сработал даже fallback-драйвер `localforage`), `getAuthItem('sre_user_registered')` бросает исключение → в `router.beforeEach` промис гарда отклоняется, `next()` никогда не вызывается, навигация зависает без обработки ошибки; в `App.vue` необработанное отклонение промиса внутри `onMounted(async () => { await initializeApp() })` прерывает инициализацию до строки, где выставляется `isAuthenticated`. То есть именно в целевом для этого тикета крайнем случае (проблемы с storage на iOS) поведение приложения становится **хуже**, чем было до миграции (раньше в этом месте практически ничего не могло бросить исключение).

**Исправление:** обернуть обе строки в `try/catch` с безопасным fallback (например, трактовать ошибку чтения как `isRegistered = false` или явный редирект на `/login`, аналогично тому, как уже сделан catch для `supabase.auth.getSession()` в этих же функциях).

### 2. Обязательные по DoD ручные проверки не выполнены

`IMPLEMENTATION_REPORT.md` прямо указывает: *"Не выполнялись в рамках данной реализации (нет доступа к браузеру/iOS-устройству в среде разработки)"* — включая:
- Подтверждение самой read-through-миграции (`localStorage` → `LocalForage`) на реальных ранее сохранённых данных — это тот самый пункт, который `IMPLEMENTATION_PLAN.md` в разделе "Риски" называет решающим ("если read-through-логика в адаптере реализована с ошибкой, все текущие залогиненные пользователи будут разом разлогинены при следующем визите... требует обязательной проверки ... перед мерджем").
- iOS Safari (обычная вкладка и standalone "Добавлено на экран Домой").
- Регрессия на Android.

Код-ревью (построчная сверка + чистая сборка) подтверждает, что реализация *соответствует плану*, но не подтверждает, что задача тикета *фактически решена* — цель тикета сформулирована как наблюдаемый результат ("исключить случаи неожиданного разлогинивания на iOS"), а не только как рефакторинг кода. Без хотя бы базовой ручной проверки миграции существующих данных нельзя исключить регресс уровня "все пользователи разлогинены после деплоя", который сам план называет главным риском.

Не блокирует ревью кода как такового (сам код корректен), но блокирует мердж/релиз до выполнения — вынесено в "Список обязательных исправлений".

## Minor

### 3. Устаревшие комментарии/текст, оставшиеся от удалённого пароль-fallback

- `src/views/LoginView.vue:14` — HTML-комментарий `<!-- Auto-login spinner -->` и текст `"Automatische Anmeldung..."` (строка 19): раньше спиннер действительно мог означать автологин по сохранённому паролю, теперь под ним скрывается только проверка активной сессии (`supabase.auth.getSession()`), автологина по паролю больше нет. Текст вводит пользователя в заблуждение (пусть и на доли секунды).
- `src/App.vue:232` — JSDoc-комментарий `* Checks for existing session or attempts auto-login` над `initializeApp()` — вторая часть фразы больше не соответствует действительности, функция больше не "attempts auto-login".

Не влияет на функциональность, но стоит поправить формулировки, раз уж концепция auto-login-по-паролю полностью удалена — иначе комментарии будут вводить в заблуждение следующего разработчика.

### 4. Небольшой прирост задержки на каждой навигации

`router/index.js:150` теперь делает асинхронное обращение к IndexedDB (`getAuthItem`) на **каждый** вызов `router.beforeEach`, то есть на каждую навигацию по приложению, а не только при логине — раньше это было синхронное чтение `localStorage` без какой-либо задержки. В абсолютных цифрах пренебрежимо мало (IndexedDB-чтение — единицы мс, к тому же guard и до этого делал сетевой вызов `supabase.auth.getSession()` для защищённых маршрутов), но для гостевых/публичных страниц это новая асинхронная операция там, где её раньше не было вообще. Не требует исправления, отмечено по запросу "проверить проблемы производительности".

# Архитектурные замечания

- Модуль `src/modules/storage.js` реализован в точности так, как специфицировал план: `authLocalForage` как отдельный `createInstance`, key-agnostic passthrough-адаптер, единая точка `getAuthItem`/`setAuthItem`/`removeAuthItem`. Хорошее разделение ответственности, никакой логики миграции не размазано по вызывающим местам.
- Разработчик самостоятельно улучшил спецификацию плана: `removeAuthItem`/`removeItem` теперь **всегда** дополнительно чистит legacy-ключ в `localStorage`, даже если ключ не мигрировался через `getItem` — в `REVIEW_ARCH_REPORT.md` (Minor #2) это было отмечено как недоспецифицированное поведение плана; в реализации это закрыто консервативно и правильно.
- Комментарии в новом файле — на немецком (`Persistente Speicherung...`, `Liest einen Wert...`), соответствует требованию `readme.md` ("NO Russian words anywhere in code, comments, or UI text"). В `REVIEW_ARCH_REPORT.md` был риск, что новый файл продолжит существующий паттерн нарушения (как в `stores/config.js`) — этого не произошло.
- Единственный пробел, не закрытый архитектурно — обработка ошибок storage-слоя (Major #1 выше). Это ровно то, что `REVIEW_ARCH_REPORT.md` отмечал как Minor #3 ("не описан error-handling storage-слоя при отказе всех драйверов LocalForage") — на этапе реализации оказалось, что пробел не только в спецификации плана, но и в самом коде, причём в двух конкретных, критичных для инициализации точках входа.
- Version pinning `localforage`: добавлен как `"^1.10.0"`, тем же стилем `^`-диапазона, что и все прочие зависимости проекта (`@supabase/supabase-js`, `pinia` и т.д.) — соответствует существующей практике репозитория, отдельно фиксировать версию не обязательно.
