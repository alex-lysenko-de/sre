# IMPLEMENTATION_PLAN — Ticket 105

## Тикет
`https://sre-nrw.de/login` — Migration of authentication storage from Local Storage to LocalForage.

Проблема: вся информация об авторизации (сессия Supabase, флаг регистрации, сохранённые credentials для автовхода, кеш профиля пользователя) хранится исключительно в `window.localStorage`. На iOS (Safari-вкладка и standalone-режим "Добавлено на экран Домой" используют разные storage-контейнеры; iOS может выгружать localStorage под давлением на диск и по правилам ITP) это приводит к периодической потере сессии и принудительному повторному логину. На Android такого не наблюдается.

*(Примечание: `CLAUDE.md` в репозитории не найден — план основан на `readme.md` и текущем коде.)*

---

## Цель

Перевести персистентность данных авторизации с `localStorage` на `LocalForage` (IndexedDB с автоматическим fallback), чтобы:
1. Сессия Supabase (access/refresh token) переживала перезапуск приложения и добавление на экран "Домой" на iOS так же надёжно, как на Android.
2. Флаг "пользователь зарегистрирован" и кеш профиля пользователя хранились в том же надёжном хранилище.
3. Устранить существующий security-риск — хранение пароля пользователя в обратимом виде (`base64`) в `localStorage`.

Миграция должна быть прозрачной для уже вошедших пользователей (без принудительного релогина сразу после деплоя) и не менять архитектурные паттерны проекта (Composables + Pinia + модульный доступ к внешним сервисам, см. `readme.md`).

## Изменения в существующих модулях

### `src/supabase.js`
`createClient(supabaseUrl, supabaseKey)` сейчас вызывается без опций → SDK использует `window.localStorage` по умолчанию (`persistSession: true`). Меняем на `createClient(supabaseUrl, supabaseKey, { auth: { storage: supabaseStorageAdapter } })`, где `supabaseStorageAdapter` — объект из нового модуля (см. "Новые компоненты"), реализующий интерфейс `SupportedStorage` (`getItem`/`setItem`/`removeItem`, все — асинхронные, что `@supabase/supabase-js` v2 поддерживает "из коробки"). Ключ, под которым SDK хранит токен (`sb-<ref>-auth-token`), не меняется и не должен жёстко прописываться в адаптере — адаптер должен быть key-agnostic passthrough.

### `src/App.vue`
- `initializeApp()` (строка 234): синхронное чтение `localStorage.getItem('sre_user_registered')` заменяется на `await getAuthItem('sre_user_registered')` из нового модуля. Функция уже `async`, изменение чисто механическое.
- `attemptAutoLogin()`, `getSavedCredentials()`, `clearSavedCredentials()` (строки 269–355) — **удаляются целиком** как часть решения по `auth_credentials` (см. ниже "Архитектурное решение по паролю"). `logout()` перестаёт вызывать `clearSavedCredentials()`.
- `checkAuth()` — без изменений (использует только `supabase.auth.getSession()`, которая уже проходит через новый адаптер прозрачно).

### `src/router/index.js`
Строка 149: `localStorage.getItem('sre_user_registered')` → `await getAuthItem('sre_user_registered')`. Guard уже объявлен как `async (to, from, next) => {...}` — изменение локальное, порядок и логика веток не меняются.

### `src/views/LoginView.vue`
- Убираются: `rememberMe` (чекбокс "Angemeldet bleiben"), `getSavedCredentials()`, `saveCredentials()`, `clearCredentials()`, `autoLogin()` и связанный с ними блок в `onMounted` — по той же причине, что и в `App.vue`.
- `onMounted` продолжает проверять `supabase.auth.getSession()` для решения "уже есть активная сессия → сразу редирект" — эта часть логики авто-входа **остаётся**, просто перестаёт иметь пароль-fallback.
- `handleLogin()`: `localStorage.setItem('sre_user_registered', 'true')` (строка 213) → `await setAuthItem('sre_user_registered', 'true')`. Функция уже `async`.

### `src/views/WelcomeView.vue`
`localStorage.setItem('sre_user_registered', 'true')` (строка 206) → `await setAuthItem('sre_user_registered', 'true')`. `handleRegister()` уже `async`.

### `src/composables/useUser.js`
`loadFromCache`, `saveToCache`, `clearCache` переписываются на использование `getAuthItem`/`setAuthItem`/`removeAuthItem` вместо прямых вызовов `localStorage.*`. Все три становятся `async` (сейчас синхронные).

### `src/stores/user.js`
Все места, вызывающие `user.loadFromCache()` / `user.saveToCache(...)` / `user.clearCache()` (строки 91, 102, 139, 170, 203, 64), добавляют `await`. Все затронутые actions (`loadUser`, `assignUserToGroup`, `assignUserToBus`, `updateUserPresence`, `clearUserCache`) уже `async`-функции (кроме `clearUserCache`, которую нужно сделать `async`) — изменение механическое, без пересмотра бизнес-логики.

## Новые компоненты

### `src/modules/storage.js` (новый модуль)
Единая точка доступа к LocalForage для авторизационных данных, в соответствии с принятым в проекте разделением "modules/ → Config, helpers, global utilities" (см. `readme.md`, "Projektstruktur"). Экспортирует:

- `authLocalForage` — сконфигурированный instance: `localforage.createInstance({ name: 'sre-app', storeName: 'auth' })`. Отдельный instance (а не `localforage` по умолчанию) — чтобы не конфликтовать с возможными будущими LocalForage-хранилищами других доменов (config, кеш детей и т.д.).
- `supabaseStorageAdapter` — объект `{ getItem, setItem, removeItem }`, передаваемый в `createClient()`. `getItem(key)` реализует **read-through lazy-миграцию**: сначала ищет значение в `authLocalForage`; если не найдено — проверяет legacy `window.localStorage.getItem(key)`; если там есть значение — копирует его в `authLocalForage`, удаляет из `window.localStorage` (чтобы не оставлять токен/пароль в старом хранилище) и возвращает. Если нет нигде — `null`.
- `getAuthItem(key)`, `setAuthItem(key, value)`, `removeAuthItem(key)` — обобщённые async-хелперы с той же read-through-миграцией, используемые для собственных ключей приложения (`sre_user_registered`, `user_info_cache`), чтобы не дублировать логику миграции в каждом файле.

**Почему read-through, а не отдельный шаг миграции при старте:** `@supabase/supabase-js` инициализирует и пытается восстановить сессию сразу при вызове `createClient()` (на этапе импорта `src/supabase.js`, до `app.mount()`). Если делать "миграцию одним махом" в `main.js`, возникает риск гонки — клиент может успеть прочитать (пустое) новое хранилище раньше, чем миграция скопирует туда старые данные. Ленивая миграция "на чтение" встроена в сам адаптер и не зависит от порядка инициализации — какой бы модуль ни запросил ключ первым, он получит корректно мигрированное значение.

### Архитектурное решение по `auth_credentials` (пароль-fallback)

Сейчас `auth_credentials` хранит `base64(JSON({email, password}))` — это не шифрование, а тривиально обратимое кодирование, и по сути это дублирующий, менее надёжный механизм персистентности сессии, изначально появившийся как компенсация ненадёжности `localStorage`. После перехода на LocalForage сам смысл этого fallback-а исчезает: устойчивость сессии обеспечивается refresh-токеном Supabase, который теперь тоже хранится в LocalForage.

**Решение: механизм "сохранить пароль и перелогиниться при отсутствии сессии" удаляется полностью** (чекбокс "Angemeldet bleiben" в `LoginView.vue`, `auth_credentials`, весь код в `App.vue`/`LoginView.vue`, связанный с ним). Единственным механизмом восстановления сессии остаётся штатный persistSession/refreshToken самого Supabase SDK поверх LocalForage.

Это меняет видимое поведение UI (пропадает чекбокс) — решение зафиксировано здесь как часть архитектуры тикета, но требует явного подтверждения на review/QA перед реализацией, так как это осознанный отказ от текущей (пусть и небезопасной) функциональности, а не просто рефакторинг хранилища.

## UI изменения

- Для пользователя внешне ничего "нового" не появляется — миграция направлена на то, чтобы существующее поведение ("не разлогинивается при перезапуске") стало наконец стабильным на iOS.

### Почему удаление чекбокса не отключает сохранение входа, а наоборот

Важно не спутать причину и следствие: сегодня в коде работают **два независимых механизма**, и чекбокс управляет только вторым, вспомогательным.

1. **Сессия Supabase** (access + refresh token) — сохраняется **всегда**, чекбокс на неё не влияет вообще ни сейчас, ни после миграции. Это и есть основной механизм "не нужно вводить пароль при повторном заходе". Он уже работает сегодня — но ненадёжно на iOS именно из-за `localStorage`, что и чинит этот тикет.
2. **`auth_credentials`** — отдельно сохранённые email+пароль (почти в открытом виде), которые используются только как аварийный fallback: если сессия из п.1 вдруг пропала, приложение тайно от пользователя повторно логинится этим сохранённым паролем. Именно этим управляет чекбокс — не "запоминать меня или нет", а "хранить ли ещё и пароль в открытом виде на случай, если основной механизм сломается".

Пункт 2 — костыль, появившийся именно потому, что пункт 1 был ненадёжен на `localStorage`/iOS. После переезда сессии (п.1) на LocalForage она сама становится надёжной, костыль теряет смысл и убирается вместе с чекбоксом, который им управлял.

**До миграции:**

| Состояние | Поведение |
|---|---|
| Сессия Supabase жива в `localStorage` | Вход не требуется (работает, но ненадёжно на iOS) |
| Сессия потеряна, но `rememberMe` был включён и `auth_credentials` сохранён | Приложение тайно логинится заново сохранённым паролем |
| Сессия потеряна, `rememberMe` не был включён | Требуется ручной ввод email+пароля |
| Пароль нигде явно не хранится "постоянно" — зависит от галочки, поставленной при последнем логине | — |

**После миграции:**

| Состояние | Поведение |
|---|---|
| Сессия жива в LocalForage (LocalForage надёжнее на iOS, но 100% гарантии платформа не даёт — см. "Риски") | Вход не требуется, всегда, без всякой галочки |
| Сессия отсутствует (новое устройство/браузер, сброс устройства, очистка данных сайта, редкий остаточный случай потери LocalForage на iOS) | Требуется ручной ввод email+пароля — ровно один раз на этом устройстве, дальше снова автоматически |
| Пароль пользователя нигде на клиенте не хранится | Единственный способ восстановить доступ без сессии — ввести email+пароль заново; это ожидаемо и не является багом |

Email и пароль в любом случае остаются универсальным способом входа с **любого** устройства (проверяются на сервере Supabase Auth) — сохранённая локально сессия лишь ускоряет повторный вход **на том же** устройстве и никак не заменяет и не блокирует обычный логин.

## Риски

- **Асинхронность — самый большой риск.** LocalForage асинхронен там, где раньше `localStorage` был синхронным. Пропущенный `await` в любом из ~7 затронутых файлов не упадёт с ошибкой типа, а тихо вернёт `Promise` вместо строки (например, `isRegistered` всегда truthy) — нужен целенаправленный код-ревью/грep по всем изменённым файлам на предмет "забытого await", а не только ручное QA.
- **Удаление пароль-fallback — поведенческое, не только техническое решение.** Нужно подтверждение перед реализацией (см. раздел выше), так как часть пользователей может ожидать конкретно чекбокс "Angemeldet bleiben".
- **iOS не даёт формальных гарантий персистентности даже для IndexedDB** — миграция снижает риск потери сессии (IndexedDB на практике переживает большинство сценариев, из-за которых страдает `localStorage`, в частности изоляцию Safari-вкладка/standalone решается тем, что оба контекста теперь ходят в один и тот же IndexedDB origin storage… но принципиальная изоляция storage между Safari-вкладкой и standalone-иконкой на iOS остаётся архитектурным ограничением платформы и никаким клиентским хранилищем не устраняется полностью). Это нужно явно отразить в тикете/тестах: миграция уменьшает частоту проблемы, но не гарантирует 100% устранение при сценарии "зарегистрировался в Safari → добавил на Домой".
- **`localforage` в приватном режиме Safari / браузерах с отключённым IndexedDB** автоматически откатывается на WebSQL/`localStorage`-драйвер — в этом крайнем случае поведение не хуже сегодняшнего, но и не лучше; не считать это регрессией.
- **Первый деплой после миграции** — критичный момент: если read-through-логика в адаптере реализована с ошибкой, все текущие залогиненные пользователи будут разом разлогинены при следующем визите. Это единственный сценарий с широким блэст-радиусом — требует обязательной проверки по п. 10 последовательности реализации перед мерджем.
- Изменения затрагивают код, общий для всего приложения (`App.vue`, `router/index.js`, `stores/user.js`) — регресс здесь потенциально влияет на все страницы, а не только на `/login`/`/welcome`, поэтому регрессионное тестирование не может ограничиться только страницами авторизации.
