## Minor

### 1. Остаточный крайний случай: сбой сети при `userStore.loadUser()` после того, как router guard уже подтвердил admin

Если `userStore.loadUser()` в `ConfigView.vue` (новый код, `onMounted`) завершится с ошибкой (например, кратковременный сбой сети при запросе к `users`), `userInfo.role` останется `null`, но `roleChecked` всё равно выставится в `true` (т.к. `loadUser()` перехватывает исключение внутри себя и всегда выполняет `finally { this.loading = false }`, не пробрасывая ошибку наружу). В этом редком случае пользователь увидит "Sie haben keine Rechte..." вместо, например, сообщения об ошибке сети — при том, что router guard свежим запросом уже подтвердил, что он admin. Это отличается от исходного бага (тот был систематическим race condition, этот — истинный сетевой сбой), поэтому не блокирует, но стоит иметь в виду. Не требует исправления в рамках этого тикета.

### 2. `alert()` для обратной связи об ошибках

Использование `alert()` в `saveConfig()` для сообщений об ошибках — не самый современный UX-паттерн, но полностью соответствует уже принятому в проекте стилю (`router/index.js` использует `alert()` для похожих сообщений, например "Zugriff verweigert" и "Konto ist deaktiviert"). Замечание не по существу, оставлено для полноты.

# Архитектурные замечания

- Устранение `configStore.isAdmin` и перенос ответственности за проверку прав на `userStore.isAdmin` — архитектурно верно и последовательно доведено до конца.
- Новая логика `roleChecked` в `ConfigView.vue` корректно закрывает найденную ранее гонку между router guard (источник №1, всегда свежий) и `userStore.loadUser()` (источник №2, асинхронный, вызывается из `App.vue.onMounted`):
  - Проверено чтением кода Vue lifecycle: дочерние компоненты (в т.ч. компонент, отрендеренный через `router-view`) монтируются **до** `onMounted` родителя (`App.vue`). Значит, при прямом заходе на `/config` (полная перезагрузка страницы — именно сценарий из тикета) `ConfigView.onMounted` выполняется раньше, чем `App.vue.onMounted` успевает запустить `userStore.loadUser()`. Новый код корректно обрабатывает и эту ветку (сам инициирует `loadUser()`, если никто ещё не начал), и ветку конкурентной загрузки, инициированной откуда-то ещё (`watch(() => userStore.loading, ...)`, чтобы не дублировать запрос — учтено, что повторный вызов `loadUser()` является no-op благодаря `if (this.loading) return` в `stores/user.js`).
  - Гонка не может привести к бесконечному "зависанию" спиннера: `userStore.loading` гарантированно становится `false` в блоке `finally` метода `loadUser()` независимо от успеха/ошибки — соответственно `roleChecked` в любом случае будет выставлен.
  - Итоговый эффект: вместо мгновенного ошибочного "keine Rechte" пользователь на холодном старте увидит короткий спиннер "Berechtigungen werden geprüft...", который корректно сменится на реальный контент, как только `userStore` действительно наполнится ролью. Это именно то поведение, которое требовал DoD плана.
- Миграция `doc/database_migration_config_rls.sql` по структуре и стилю (`EXISTS (SELECT 1 FROM public.users WHERE users.user_id = auth.uid() AND users.role = 'admin' AND users.active = true)`) точно повторяет уже существующий в репозитории паттерн для `c_bands`/`groups` в `doc/database_migration.sql` — консистентно с остальной кодовой базой. Осознанное отличие (публичный `SELECT` вместо `TO authenticated`) обосновано и подтверждено чтением `InfoView.vue`/`router/index.js` (`/info` — публичная страница, читает `public_phone_number`).
- `saveConfig()` теперь обёрнут в `try/catch` с обратной связью пользователю — устраняет ранее найденный тихий no-op и необработанное исключение.

# Рекомендации

1. **Перед релизом обязательно применить `doc/database_migration_config_rls.sql` к боевому Supabase** (SQL Editor или `supabase db push`) и подтвердить активные политики через `pg_policies` / dashboard. Это единственный оставшийся шаг, закрывающий Critical #1 по факту, а не только в репозитории. Без этого шага риск (запись в `config` от не-admin через прямой вызов API) формально всё ещё не закрыт — миграция в репозитории сама по себе ничего не защищает, пока не выполнена в БД.
2. Выполнить ручную регрессию по актуальному чек-листу в `IMPLEMENTATION_REPORT.md` ("Ручные проверки", п.1–7) хотя бы один раз перед релизом — особенно холодный старт `/config` с throttled network под обоими тестовыми admin-аккаунтами, чтобы вживую подтвердить, что спиннер `roleChecked` действительно решает исходную проблему тикета.
3. Опционально (не блокирует): рассмотреть отдельно от этого тикета более явную обработку сетевого сбоя `loadUser()` в `ConfigView` (Minor #1) — например, отдельное сообщение "Fehler beim Prüfen der Berechtigungen, bitte neu laden" вместо "keine Rechte" при `userStore.error !== null`.
