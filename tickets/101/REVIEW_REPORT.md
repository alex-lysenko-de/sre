# Найденные проблемы

## Critical

### 1. Обязательный шаг плана (аудит/фикс RLS для таблицы `config`) пропущен, а клиентская проверка прав всё равно удалена — потенциальная дыра в безопасности

План (раздел "Риски", "Последовательность реализации" п.1–2) явно требовал: сначала подтвердить/поставить RLS-политики для таблицы `config` в Supabase, **и только потом** убирать клиентскую проверку `isAdmin` в `updateConfig()` — именно чтобы не создать окно, когда единственной защитой от записи неавторизованным пользователем остаётся легко обходимая проверка на клиенте.

В реализации это условие не выполнено:
- В `src/stores/config.js` (`updateConfig`) серверный/клиентский guard `if (!this.isAdmin) throw ...` удалён полностью.
- Единственная оставшаяся проверка — в `src/views/ConfigView.vue:76` (`if (!userStore.isAdmin) return`) — это чисто UI-уровень: обходится тривиально прямым вызовом Supabase JS SDK из консоли браузера или curl/Postman с валидным JWT любого `authenticated`-пользователя.
- Аудит RLS не выполнен (сам разработчик подтверждает это в `IMPLEMENTATION_REPORT.md`, раздел "Отклонения от плана").
- Проверка репозитория (`doc/database_migration.sql`, `doc/backup.sql`, `doc/db_triggers.sql`, `doc/users.sql`) подтверждает: таблицы `config` там нет вообще — ни `CREATE TABLE`, ни `ENABLE ROW LEVEL SECURITY`, ни `CREATE POLICY`. Т.е. RLS-статус этой таблицы задокументированной защиты не имеет, а полагаться на "наверное, там что-то настроено вручную в Supabase dashboard" — не должно быть основанием для мержа security-чувствительного изменения.

**Итог:** если в Supabase RLS для `config` не включён (что весьма вероятно, учитывая полное отсутствие следов в SQL-артефактах репозитория), после этого изменения **любой залогиненный пользователь** (не обязательно admin) сможет изменить `public_phone_number`, `base_url`, `ankunftszeit`/`abfahrtszeit` и т.д. напрямую через Supabase API, полностью в обход UI. Это прямое нарушение требования DoD плана ("Функциональность сохранения конфигурации... корректно отклоняется на уровне БД для не-admin (проверить прямым вызовом Supabase API)") — пункт не выполнен и не проверен.

**Рекомендация:** не мержить это изменение, пока не подтверждено (скриншотом/выгрузкой политик из Supabase dashboard или `pg_policies`) наличие рабочей RLS-политики на `INSERT/UPDATE/DELETE` для `config`, ограничивающей запись ролью `admin` + `active = true`. Если политики нет — либо добавить миграцию (как и предполагал план), либо не удалять клиентскую проверку в сторе до появления серверной защиты.

## Major

### 2. Race condition не устранён полностью, а перенесён из `configStore.isAdmin` в `userStore.isAdmin`

Диагноз в плане корректно определяет 3 независимых источника "isAdmin" и указывает, что нужно свести UI-гейтинг к единственному надёжному источнику. Однако анализ фактической последовательности инициализации показывает, что второй источник (`userStore.isAdmin`) **тоже не гарантированно наполнен** к моменту первого рендера `ConfigView`:

- `src/main.js` не дожидается `router.isReady()` перед `app.mount('#app')`.
- `userStore.loadUser()` вызывается только из `App.vue` (`onMounted` → `initializeApp()` → `handleAuthentication()`), т.е. асинхронно и без какой-либо синхронизации с моментом, когда `router-view` уже отрисовал `ConfigView` после успешного прохождения guard'а (`router/index.js:172-221`).
- Guard в router (источник №1, надёжный) делает свой независимый свежий запрос и корректно пускает admin-пользователя на `/config` — но это никак не гарантирует, что к этому же моменту `userStore.userInfo.role` уже заполнен. При "холодном" старте (очищенный `localStorage`, медленная сеть — именно тот сценарий, который DoD плана явно требует тестировать) вполне возможна ситуация: guard пропустил → `ConfigView` смонтирован и сделал первый рендер → `userStore.isAdmin === false` (т.к. `userInfo.role` ещё `null`, начальное состояние) → показывается тот же "Sie haben keine Rechte...".

Разница с исходным багом: `userStore.isAdmin` — реактивный getter поверх реактивного state, поэтому как только `loadUser()` завершится, UI сам переотрисуется и ошибка пропадёт без перезагрузки страницы. Это реальное улучшение (превращает "залипающий навсегда" баг во "временное мигание, которое самовосстанавливается"), но это **не то же самое**, что утверждается в плане и отчёте — "стор уже гарантированно инициализирован к моменту рендера защищённых роутов". Это утверждение ничем в коде не обеспечено (нет `await userStore.loadUser()` ни в router guard, ни в `ConfigView.onMounted` перед снятием гейта).

**Рекомендация:** один из вариантов:
- В router guard (`src/router/index.js`) после успешной проверки роли — также убедиться, что `userStore` заполнен (например, вызвать `userStore.loadUser()` там же, раз уже есть свежие данные `userData.role` — заодно и переиспользовать этот же запрос вместо дублирующего в `App.vue`), либо
- В `ConfigView.vue` `onMounted` — если `userStore.userInfo.role === null`, дождаться `await userStore.loadUser()` перед вычислением гейта, показывая спиннер (используя уже существующий `userStore.loading`) вместо немедленного "keine Rechte".

Без этого DoD-пункт "Оба тестовых Hauptbetreuer-аккаунта успешно открывают /config при множественных холодных загрузках... без единого ложного 'keine Rechte'" не может считаться подтверждённым — а он не был протестирован вручную (см. "Ручные проверки" в отчёте, помечены как невыполненные).

### 3. Пункты DoD и обязательных ручных проверок не выполнены, но задача передана как готовая

`IMPLEMENTATION_REPORT.md` сам перечисляет невыполненные обязательные пункты (аудит RLS, регрессия race condition на холодных загрузках, проверка `saveConfig` от лица не-admin через прямой API), но при этом README/отчёт по сути описывает код как завершённый. Учитывая, что баг в тикете — нестабильный race condition, отсутствие фактической ручной регрессии (не просто `vite build`) означает, что исправление могло не сработать в проде так, как задумано (см. находку #2 выше) — это не было бы обнаружено сборкой.

## Minor

### 4. `v-if="!userStore.isAdmin"` не различает "ещё не знаем" и "точно не admin"

Один и тот же блок ошибки показывается и когда роль ещё не загружена (`userInfo.role === null`), и когда пользователь точно не admin. Это первопричина визуального "мигания" из находки #2 — семантически некорректно показывать пользователю текст отказа в доступе, когда на самом деле это ещё "неизвестно". Стоит завести отдельное состояние загрузки (`userStore.loading` уже существует и для этого подходит).

### 5. `saveConfig()` в `ConfigView.vue:75-80` тихо ничего не делает при `!userStore.isAdmin`

Если пользователь нажмёт "Speichern" в окне гонки из находки #2 (когда `isAdmin` ещё `false`, хотя пользователь реально admin), клик молча ничего не сделает — ни ошибки, ни индикации. Некритично, но ухудшает UX ровно в том сценарии, который и является предметом тикета.

### 6. Отсутствие обработки ошибок в `saveConfig`/`updateConfig` (не внесено этим изменением, но усугубляет риск #1)

`updateConfig()` бросает исключение при ошибке Supabase (например, при отказе RLS для не-admin), но `saveConfig()` в `ConfigView.vue` не оборачивает вызов в `try/catch` — при реальном срабатывании серверного RLS-отказа пользователь увидит необработанную ошибку в консоли без обратной связи в UI. Это существовало до изменения, но стоит зафиксировать, т.к. пункт DoD "прямым вызовом Supabase API от не-admin — должно отклоняться" будет наблюдаться пользователем именно через этот необработанный путь, если он попробует через UI от имени, которое перестало быть admin в середине сессии.

# Архитектурные замечания

- Устранение `configStore.isAdmin` как отдельного, независимого источника авторизации — правильное архитектурное решение, соответствует Single Responsibility и плану. Дублирующих ссылок на `configStore.isAdmin` в кодовой базе не осталось (проверено grep).
- `main.js` оставлен без изменений — формально это не нарушает букву DoD ("main.js не должен принимать решения об авторизации"), т.к. авторизационная логика ушла из `initConfigModule()`. Однако это же и есть часть причины находки #2: `main.js` по-прежнему не синхронизирует старт приложения ни с `router.isReady()`, ни с `userStore.loadUser()`, поэтому общая архитектурная цель плана — "не принимать решения об авторизации до того, как есть актуальная сессия" — выполнена только частично: перенесена ответственность, но не устранена гонка между независимыми источниками (router guard vs userStore).
- Использование `userStore.isAdmin` как defense-in-depth гейта в `ConfigView` соответствует явной рекомендации плана, но стоит заметить: план же описывает "целевой паттерн" остальных admin-страниц как полное отсутствие клиентского гейта (только router guard). `ConfigView` теперь отличается от этого паттерна (у неё есть свой клиентский гейт, у остальных — нет). Это осознанное отклонение, разрешённое самим планом, но стоит явно задокументировать в PR, чтобы не создать путаницу при будущем рефакторинге "все admin-страницы одинаковы".
- Само по себе отсутствие таблицы `config` в SQL-артефактах репозитория (`doc/*.sql`) — более широкая проблема качества документации/миграций проекта: часть схемы БД существует только в Supabase dashboard и не воспроизводима из репозитория. Не в скоупе этого тикета, но стоит завести отдельный тикет на экспорт текущей схемы `config` (structure + RLS) в `doc/database_migration.sql`.

# Рекомендации

1. Не мержить до подтверждения RLS-статуса таблицы `config` (Critical #1) — это самый важный блокер.
2. Закрыть гонку `userStore.isAdmin` явным ожиданием загрузки роли перед рендером гейта в `ConfigView`, либо переиспользовать данные, уже полученные в router guard (Major #2).
3. Выполнить фактическую ручную регрессию с холодным стартом и throttled network для обоих тестовых admin-аккаунтов, как того требует DoD плана, а не полагаться только на `vite build`.
4. Добавить `try/catch` с пользовательской обратной связью в `saveConfig()` — низкий приоритет, но недорого сделать заодно.
