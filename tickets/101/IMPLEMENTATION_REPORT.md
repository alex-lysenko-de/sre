## Выполненные изменения

### `src/stores/config.js`
- Удалено поле состояния `isAdmin` из `state()`.
- `initConfigModule()` больше не делает `supabase.auth.getUser()` и запрос `users.role` — теперь это просто `await this.loadConfig(); this.subscribeToRealtime()`. Тем самым устранён race condition (источник №3 из диагноза): конфиг-стор больше не вычисляет и не хранит "админ ли я" один раз при бутстрапе.
- В `updateConfig()` убрана клиентская проверка `if (!this.isAdmin) throw ...`. Проверка перенесена на вызывающую сторону (см. `ConfigView.vue`), как рекомендовано планом — она носит чисто UX-характер и не является security-границей.

### `src/views/ConfigView.vue`
- Гейт отображения `v-if="!configStore.isAdmin"` заменён на `v-if="!userStore.isAdmin"` — единый надёжный реактивный источник истины (тот же, что уже используется в `App.vue`/`MainView.vue`), гарантированно наполненный к моменту рендера защищённых маршрутов (`requiresAdmin: true` в router guard).
- Добавлен импорт и инстанцирование `useUserStore()`.
- В `saveConfig()` добавлена клиентская UX-проверка `if (!userStore.isAdmin) return` перед вызовом `configStore.updateConfig()` — компенсирует убранную проверку в сторе.
- Текст/разметка блока "Sie haben keine Rechte..." не изменены — согласно плану, менялся только источник данных для `v-if`.

### `src/main.js`
Изменений не потребовалось. `main.js` по-прежнему делает блокирующий `await configStore.initConfigModule()` до `app.mount()`, но так как из `initConfigModule()` полностью удалена логика авторизации (см. выше), этот блокирующий путь больше не принимает решений об авторизации — требование DoD выполнено на уровне `config.js`, без правок `main.js`, что соответствует принципу минимальных изменений.

## Отклонения от плана

- **Аудит RLS-политик таблицы `config` в Supabase (шаги 1–2 плана) не выполнен** — требует доступа к Supabase dashboard/CLI с продовыми credentials, которого нет в рамках этой сессии разработки. В репозитории (`doc/*.sql`, `supabase/`) миграций для таблицы `config` не найдено, что подтверждает диагноз плана: статус RLS для неё в коде не задокументирован.
  - Это прямо влияет на риск, описанный в плане: после удаления клиентской проверки `isAdmin` в `updateConfig()` единственной защитой от записи неавторизованными пользователями остаётся RLS на стороне БД. Если RLS для `config` не настроен, любой `authenticated`-пользователь сможет менять конфигурацию напрямую через Supabase API, минуя UI.
  - **Требуется отдельная ручная проверка перед мержем в прод** (см. "Ручные проверки" ниже).
- Composable `useAdminGuard()` (упомянутый в плане как опциональное улучшение) не создавался — план явно отмечает его необязательным для этого тикета.
- `USER_CACHE_KEY`, не привязанный к `user_id` (риск, отмеченный в плане) — не трогался, план явно выносит это в отдельный тикет.

## Требуемые миграции

Возможно потребуется миграция SQL для RLS-политик таблицы `config`, аналогичная существующим политикам для `c_bands`/`groups` в `doc/database_migration.sql`:
- `SELECT` — `authenticated` (или шире).
- `INSERT`/`UPDATE`/`DELETE` — только `users.role = 'admin' AND users.active = true`.

Миграция **не создана** в рамках этой сессии, так как требует предварительного аудита текущего состояния Supabase (нельзя писать миграцию "вслепую", не зная, есть ли уже политика). Необходимо выполнить аудит и, при необходимости, применить миграцию отдельным шагом до релиза этого изменения.

## Ручные проверки

Перед мержем/релизом обязательно выполнить (без них риск описан выше не закрыт):

1. **Аудит RLS `config`**: в Supabase dashboard проверить, включён ли RLS для таблицы `config`, и есть ли политика, ограничивающая `INSERT`/`UPDATE`/`DELETE` ролью `admin`. При отсутствии — применить миграцию (см. выше) **до** выкладки этого изменения в прод.
2. **Регрессия race condition**: несколько "холодных" загрузок `/config` (очищенный `localStorage`, throttled network в DevTools) под обоими тестовыми Hauptbetreuer-аккаунтами — ни разу не должно появиться "Sie haben keine Rechte...".
3. **`saveConfig`**: проверить, что сохранение работает для `admin`, и что прямой вызов Supabase API (`update` на таблице `config`) от имени не-`admin` пользователя отклоняется на уровне БД (не только UI).
4. **Регрессия остальных admin-страниц**: `AdminBusView`, `UsersView`, `DaysEditView`, `InviteGeneratorView`, `ChildrenView` — не были затронуты изменениями, но стоит подтвердить, что они по-прежнему работают как раньше.
5. `npx vite build` выполнен локально в рамках этой сессии — сборка проходит без ошибок (см. вывод сборки), синтаксических проблем в изменённых файлах нет. Это не заменяет функциональную/browser-проверку.

## Исправления по REVIEW_REPORT.md

### Critical #1 — RLS для `config` не подтверждён, клиентская проверка удалена

Добавлена миграция `doc/database_migration_config_rls.sql`:
- `CREATE TABLE IF NOT EXISTS public.config (...)` — идемпотентно, если таблица уже существует в Supabase с другой структурой, оператор не выполнит никаких изменений (`IF NOT EXISTS`), поэтому миграция безопасна для прогона поверх текущего прода.
- `ALTER TABLE public.config ENABLE ROW LEVEL SECURITY`.
- `SELECT` — политика `TO public` (не только `authenticated`): подтверждено чтением кода, что `/info` — полностью публичная страница (`meta: { requiresAuth: false }`, доступна незарегистрированным гостям через `router/index.js`) и рендерит `configStore.config.public_phone_number` (`InfoView.vue:25`). Ограничение `SELECT` только `authenticated` сломало бы эту страницу.
- `INSERT/UPDATE/DELETE` (`FOR ALL`) — только для `users.role = 'admin' AND users.active = true`, по образцу существующих политик для `c_bands`/`groups` в `doc/database_migration.sql`; добавлен явный `WITH CHECK` (в дополнение к `USING`) для гарантии, что и `INSERT`, и `UPDATE` проверяются одинаково строго.

**Важно:** это миграция, подготовленная по коду и по существующему в репозитории паттерну (`c_bands`, `groups`), а не подтверждённый факт текущего состояния Supabase — реальное применение (`supabase db push` / выполнение в SQL Editor) и проверка результата (`pg_policies`) остаются обязательной ручной проверкой перед мержем, см. обновлённый раздел "Ручные проверки" ниже. Без выполнения этой миграции в самой Supabase риск, описанный в Critical #1, не закрыт — файл в репозитории сам по себе ничего не защищает, пока не применён.

### Major #2 / Minor #4 — гонка `userStore.isAdmin` при холодном старте, неразличимость "неизвестно" и "не admin"

`src/views/ConfigView.vue`:
- Добавлено состояние `roleChecked` (изначально `false`).
- В `onMounted`: если `userStore.userInfo.role` уже известен — `roleChecked = true` сразу; если `userStore.loadUser()` ещё не запущен — запускается и дожидается `await`; если загрузка уже идёт (например, инициирована `App.vue`) — используется `watch(() => userStore.loading, ...)`, чтобы дождаться её завершения, не вызывая `loadUser()` повторно (повторный вызов — no-op из-за `if (this.loading) return` в `stores/user.js`, поэтому просто `await userStore.loadUser()` здесь не сработал бы корректно).
- Шаблон теперь различает три состояния: `!roleChecked` → спиннер "Berechtigungen werden geprüft...", `roleChecked && !userStore.isAdmin` → блок "keine Rechte", иначе — обычный контент. Это устраняет ложный "keine Rechte" в окне гонки и одновременно закрывает Minor #4 (раньше оба состояния — "неизвестно" и "точно не admin" — показывали один и тот же блок ошибки).

### Minor #5 / #6 — `saveConfig()` тихо no-op, нет обработки ошибок

`saveConfig()` в `ConfigView.vue`:
- При `!userStore.isAdmin` теперь показывает `alert('⛔ Sie haben keine Rechte...')` вместо тихого выхода.
- Вызов `configStore.updateConfig()` обёрнут в `try/catch`; при ошибке (в т.ч. будущем отказе RLS для не-admin) — `console.error` + `alert` с текстом ошибки, вместо необработанного исключения в консоли без обратной связи пользователю.
