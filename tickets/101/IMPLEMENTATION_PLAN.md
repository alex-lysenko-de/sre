# IMPLEMENTATION_PLAN — Ticket 101

## Тикет
При открытии `https://sre-nrw.de/config` Hauptbetreuer (роль `admin`, неограниченные права) видит:
> Sie haben keine Rechte, auf diese Seite zuzugreifen.

При этом у другого Hauptbetreuer та же страница открывается нормально. Симптом нестабильный, зависит от пользователя/сессии — типичный признак race condition, а не проблемы с данными (роль в БД у обоих пользователей одинаковая — `admin`).

---

## Цель

Устранить ошибочный отказ в доступе на странице `/config` для пользователей с ролью `admin`, убрав причину нестабильности — гонку при определении прав "администратор ли я" на старте приложения. Итоговая цель архитектурно шире одного экрана: свести проверку `isAdmin` к единому надёжному источнику истины, чтобы подобный баг не повторился на других admin-страницах и не возникал повторно при доработке `ConfigView`.

---

## Диагноз (root cause)

В проекте одновременно существуют **три независимых механизма** определения "является ли текущий пользователь администратором", которые не синхронизированы между собой:

| # | Источник | Где вычисляется | Когда обновляется | Надёжность |
|---|----------|------------------|--------------------|------------|
| 1 | Router guard (`src/router/index.js`, `beforeEach`) | Живой запрос `users.role` к Supabase | При каждой навигации | Высокая — всегда свежие данные |
| 2 | `userStore.isAdmin` (`src/stores/user.js` → `useUser`/`useSupabaseUser`) | Кэш в `localStorage` (`user_info_cache`, TTL 1 мин) | При `loadUser()`, вызывается в `App.vue` при монтировании и при каждой смене route | Средняя, но реактивна и используется корректно (меню в `App.vue`, `MainView.vue`) |
| 3 | `configStore.isAdmin` (`src/stores/config.js` → `initConfigModule()`) | Отдельный запрос `users.role` к Supabase | **Один раз**, в `main.js`, до `app.mount()`, и больше никогда | **Низкая** — см. ниже |

Именно источник **№3** гейтит рендеринг `ConfigView.vue` (`v-if="!configStore.isAdmin"` → блок "keine Rechte").

`main.js` делает блокирующий `await configStore.initConfigModule()` **до** `app.mount('#app')`. Внутри вызывается `supabase.auth.getUser()` сразу после создания клиента — на этот момент восстановление сессии Supabase из `localStorage`/обновление токена может быть ещё не завершено (асинхронный внутренний процесс клиента). Если в этот момент `getUser()` вернёт `user: null` (медленная сеть, "холодный" старт вкладки, идущий refresh токена и т.п.), то:

```js
this.isAdmin = false  // конфиг-стор.config.js:162
```

устанавливается один раз и **никогда не переоценивается** — ни при последующих успешных навигациях, ни при повторном заходе на `/config`, потому что `ConfigView.vue` не вызывает `initConfigModule()` повторно в своём `onMounted`.

Router guard, напротив, пропускает пользователя на `/config` (роль в БД действительно `admin`, проверено свежим запросом) — но затем `ConfigView` показывает ошибку, опираясь на устаревший/ошибочный флаг №3. Это объясняет, почему у одного Hauptbetreuer работает (боту "повезло" с таймингом при загрузке), а у другого — нет (гонка сработала не в его пользу). Это чистый race condition на клиенте, не связанный с данными в БД.

Дополнительно замечено (не блокирует фикс, но важно для риска):
- Таблицы `config` **нет** ни в `doc/database_migration.sql`, ни в `doc/users.sql`, ни в других SQL-артефактах — статус RLS-политик для неё неизвестен из документации. Серверная защита от записи неавторизованными пользователями не подтверждена.
- `USER_CACHE_KEY = 'user_info_cache'` в `useUser.js` не привязан к `user_id` — общий ключ `localStorage` на устройство. Не относится напрямую к тикету, но повышает риск "чужих" данных при смене пользователя на одном устройстве/браузере. Рекомендуется отдельным тикетом.

---

## Затрагиваемые модули

- `src/main.js` — точка бутстрапа приложения, блокирующая инициализация `configStore`.
- `src/stores/config.js` — источник `isAdmin` №3 (дублирующий, ненадёжный).
- `src/views/ConfigView.vue` — единственное место, где используется `configStore.isAdmin` для гейтинга UI.
- `src/stores/user.js` / `src/composables/useUser.js` — существующий надёжный, реактивный источник `isAdmin` (№2), который станет единым источником истины для UI-уровня.
- `src/router/index.js` — уже корректно реализованный guard (№1), изменений не требует, служит эталоном паттерна для остальных admin-страниц.
- Supabase: таблица `config`, её RLS-политики (требуется аудит, см. "Риски").

Для справки — остальные admin-страницы (`AdminBusView.vue`, `UsersView.vue`, `DaysEditView.vue`, `InviteGeneratorView.vue`, `ChildrenView.vue`) **не** дублируют проверку прав на уровне компонента — они полагаются исключительно на router guard и потому этим багом не затронуты. Это и есть целевой паттерн, к которому нужно привести `ConfigView.vue`.

---

## Изменения в существующих модулях

### `src/views/ConfigView.vue`
- Убрать зависимость от `configStore.isAdmin` как гейта отображения.
- Привести компонент к паттерну остальных admin-страниц: полагаться на `router.beforeEach` (`meta: { requiresAdmin: true }`) как единственный authorization-гейт для *доступа к странице*.
- Если требуется defense-in-depth на уровне компонента (например, чтобы избежать "мигания" формы до того, как guard отработает, или для консистентности с `MainView.vue`) — использовать `userStore.isAdmin` (реактивный getter, уже надёжно наполняется в `App.vue`), а не `configStore.isAdmin`.
- Блок "Sie haben keine Rechte..." либо убрать полностью (раз guard уже не пускает не-админов на роут), либо оставить как fallback на `userStore.isAdmin` — решение зависит от того, хотим ли мы UX-подстраховку. Рекомендация: оставить fallback на `userStore.isAdmin`, т.к. это дешёво и не создаёт новой гонки (стор уже гарантированно инициализирован к моменту рендера защищённых роутов).

### `src/stores/config.js`
- Удалить поле состояния `isAdmin` и его вычисление внутри `initConfigModule()` — обязанность "знаю ли я, админ ли текущий пользователь" не должна лежать на `config`-сторе (нарушение Single Responsibility: config-стор должен отвечать только за данные конфигурации, а не за авторизацию).
- `initConfigModule()` упрощается до загрузки конфигурации и подписки на realtime — без запроса к `users`.
- `updateConfig()` — убрать клиентскую проверку `if (!this.isAdmin) throw ...`, заменить (или дополнить) проверкой через `useUserStore().isAdmin` на вызывающей стороне (в `ConfigView.vue`, перед вызовом `saveConfig`), либо оставить только серверную защиту через RLS (см. "Изменения БД"). Клиентская проверка носит чисто UX-характер и не является security-границей.

### `src/main.js`
- Убрать блокирующий `await configStore.initConfigModule()` до `app.mount()` в части, связанной с определением роли (сама загрузка конфигурации может остаться, т.к. она не завязана на сессию пользователя и не должна давать sistema race conditions — но нужно убедиться, что `loadConfig()`/`fetchFromSupabase()` не завязаны на RLS-политики, требующие авторизации; если `config` — публично читаемая таблица, blocking-await можно оставить только для данных, не для `isAdmin`).
- Цель: приложение больше не должно принимать решения об авторизации до полного маунта и до того, как `userStore`/router успели получить актуальную сессию.

## Новые компоненты

Новых компонентов, сторов или composables не требуется. Задача — устранить дублирование существующей логики, а не добавить новую.

(Опционально, если появится третья/четвёртая admin-страница с похожей потребностью в UI-гейте: можно вынести `const isAdmin = computed(() => userStore.isAdmin)` в переиспользуемый composable, например `useAdminGuard()`, чтобы не размножать `computed` по компонентам вручную. Не обязательно для этого тикета — фиксируется как возможное улучшение, не как требование.)

## Изменения БД

Требуется **аудит**, не миграция "вслепую" (миграция без проверки текущего состояния Supabase рискованна):

1. Проверить в Supabase dashboard, включён ли RLS для таблицы `config` и есть ли политика, ограничивающая `UPDATE`/`INSERT`/`DELETE` ролью `admin` (по аналогии с политиками для `c_bands`/`groups` в `doc/database_migration.sql`).
2. Если политики нет — добавить её отдельной миграцией:
   - `SELECT` — разрешён `authenticated` (или шире, если конфиг должен быть публично читаем, например для страницы `/info`).
   - `INSERT`/`UPDATE`/`DELETE` — только для пользователей с `users.role = 'admin' AND users.active = true`, тем же паттерном, что уже используется для `c_bands`/`groups` в существующих миграциях.
3. Изменений в схеме `users`/`c_bands`/`children`/`scans` и т.д. это тикет не требует.

## API изменения

Изменений в Supabase Edge Functions (`auth`, `delete-user`, `invite-accept`, `invite-generate`) не требуется — они не участвуют в этом сценарии. Единственное "API"-взаимодействие, которое меняется — количество и место клиентских запросов `users.role`:
- Убирается один избыточный запрос `users.role` (ранее делался в `configStore.initConfigModule()`).
- Остаётся запрос в router guard (обязателен, это единственная серверная проверка на клиенте) и запрос в `userStore.loadUser()` (уже существует, переиспользуется).

## UI изменения

- `ConfigView.vue`: блок "Sie haben keine Rechte, auf diese Seite zuzugreifen." либо убирается, либо начинает опираться на `userStore.isAdmin` вместо `configStore.isAdmin`. Внешний вид/текст не меняется — меняется только источник данных для `v-if`.
- Никаких изменений в навигации (`App.vue`) не требуется — там уже используется корректный `userStore.isAdmin`.

## Последовательность реализации

1. **Аудит Supabase**: проверить текущие RLS-политики таблицы `config` (см. "Изменения БД", п.1). Зафиксировать результат до начала кодинга — от этого зависит, нужна ли миграция.
2. Если RLS отсутствует/недостаточен — подготовить и применить SQL-миграцию с политиками для `config`.
3. `src/stores/config.js`: удалить поле `isAdmin` и связанный код в `initConfigModule()` / `updateConfig()`.
4. `src/main.js`: скорректировать вызов `configStore.initConfigModule()` (убрать любую логику авторизации из пути, блокирующего маунт).
5. `src/views/ConfigView.vue`: заменить гейт `configStore.isAdmin` → `userStore.isAdmin`; при сохранении (`saveConfig`) добавить клиентскую проверку `userStore.isAdmin` перед вызовом `updateConfig()` (UX-подсказка, не security-граница).
6. Ручная регрессия (см. Definition of Done) — воспроизвести исходный баг было нельзя стабильно (race condition), поэтому проверка должна включать многократные "холодные" загрузки (очистка кэша/сессии, throttled network) для обоих тестовых admin-аккаунтов.
7. Проверить, что остальные admin-страницы (`AdminBusView`, `UsersView`, `DaysEditView`, `InviteGeneratorView`, `ChildrenView`) не затронуты и продолжают работать как раньше (regression, не должны были меняться).

## Риски

- **RLS-статус таблицы `config` неизвестен из репозитория.** Если RLS не настроен и таблица открыта на запись для любого `authenticated`-пользователя, удаление клиентской проверки `configStore.isAdmin` без замены её реальной серверной политикой **создаст дыру в безопасности** (любой залогиненный пользователь сможет менять конфигурацию через прямой вызов Supabase API, минуя UI). Порядок в "Последовательность реализации" (аудит/миграция — шаг 1–2, до правок клиента) специально выстроен так, чтобы исключить этот риск.
- **Race condition мог маскировать другие проблемы** с восстановлением сессии Supabase при cold start (например, если `attemptAutoLogin()` в `App.vue` тоже завязан на похожий тайминг). Стоит держать в уме при регрессионном тестировании, хотя это не входит в scope тикета 101.
- Удаление `configStore.isAdmin` — breaking change для любого кода, который мог начать полагаться на него позже (на момент анализа таких мест не найдено — grep показал использование только в `ConfigView.vue`), но стоит перепроверить перед мержем на актуальном `main`.
- Если `loadConfig()`/`fetchFromSupabase()` в `config.js` требуют авторизации (RLS на `SELECT` тоже ограничен), блокирующий `await` в `main.js` может по-прежнему создавать похожую гонку для отображения самих данных конфигурации (не прав доступа) — нужно проверить отдельно и, при необходимости, вынести и эту загрузку из блокирующего пути бутстрапа.

## Definition of Done

- [ ] Подтверждён (и при необходимости исправлен) статус RLS-политик таблицы `config` в Supabase.
- [ ] `configStore` больше не хранит и не вычисляет `isAdmin`.
- [ ] `ConfigView.vue` использует `userStore.isAdmin` (либо полностью полагается на router guard, без собственного гейта) — единый источник истины для UI-уровня, идентичный паттерну остальных admin-страниц.
- [ ] `main.js` не принимает решений об авторизации в блокирующем пути до `app.mount()`.
- [ ] Оба тестовых Hauptbetreuer-аккаунта успешно открывают `/config` при множественных "холодных" загрузках (очищенный `localStorage`, эмуляция медленной сети через DevTools throttling), без единого ложного "keine Rechte".
- [ ] Функциональность сохранения конфигурации (`saveConfig`) работает для `admin` и корректно отклоняется на уровне БД для не-`admin` (проверить прямым вызовом Supabase API, не только через UI).
- [ ] Остальные admin-страницы (`AdminBusView`, `UsersView`, `DaysEditView`, `InviteGeneratorView`, `ChildrenView`) не регрессировали.
- [ ] Изменения задокументированы в commit message / PR description с явной ссылкой на root cause (race condition в `initConfigModule()`), чтобы не повторить паттерн при добавлении новых admin-страниц в будущем.
