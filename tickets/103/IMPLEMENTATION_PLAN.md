# IMPLEMENTATION_PLAN — Ticket 103

## Тикет
`https://sre-nrw.de/scanner`

1. На Android выбирается корректная тыловая камера, на iPhone — фронтальная (селфи), которая для сканирования QR-кодов не подходит. Нужен либо надёжный автовыбор тыловой камеры, либо ручной выбор камеры из списка.
2. Звук при успешном/неуспешном скане слишком тихий, непонятно, произошло ли сканирование. Нужно заметно увеличить громкость и/или добавить вибрацию, а также усилить визуальную обратную связь — при скане менять цвет рамки на градиентные "радужные" цвета на время до 1 секунды с последующим возвратом к исходному состоянию.

---

## Цель

Сделать обратную связь сканера (выбор камеры + сигнал об успешном/неуспешном скане) одинаково надёжной на Android и iOS, без полагания на признаки/API, которые на iOS работают иначе или не работают вовсе. Реализация должна опираться на семантические, кросс-платформенные механизмы, а не на эвристики (парсинг текстовых лейблов камер, предположения о поведении звука/вибрации), которые уже один раз подвели именно на iPhone.

---

## Диагноз (root cause)

### Проблема 1 — выбор камеры

`src/views/ScannerView.vue`, `initScanner()` (строки 205-229):

```js
const devices = await Html5Qrcode.getCameras()
const backCamera = devices.find(device =>
    device.label.toLowerCase().includes('back') ||
    device.label.toLowerCase().includes('rear') ||
    device.label.includes('0')
) || devices[0]
CAMERA_ID = backCamera.id
```

Камера выбирается парсингом человекочитаемого `device.label` из `enumerateDevices()`. Это не стандартизированное, платформенно-зависимое значение:
- На Android/Chrome лейблы обычно содержат "back"/"front" — эвристика случайно работает.
- На iOS Safari (весь браузерный движок на iOS — WebKit, включая сторонние "браузеры") лейблы камер часто generic ("Camera 1", "Camera 2") или содержат технические идентификаторы. Условие `.includes('0')` в этом случае — очень слабое и может ложно сработать на произвольной подстроке в лейбле фронтальной камеры. Если ни одно условие не совпало, код берёт `devices[0]` — а на iOS первым в списке нередко оказывается именно фронтальная камера.

Итог: логика выбора камеры опирается на недокументированное, непереносимое поведение конкретных браузеров вместо семантического API.

### Проблема 2 — слабая обратная связь при скане

`playSound()` (строка 143-151) уже выставляет `sound.volume = 1.0` (максимум) — то есть громкость на уровне Web Audio API уже предельная, "увеличить громкость" программно через `volume` дальше некуда. Причина субъективной "тихости":
1. Сам звуковой файл — короткий встроенный base64 WAV с низкой амплитудой мастеринга; при 1.0 volume он всё равно звучит тихо/невыразительно.
2. **Критично для iOS**: Safari на iPhone по умолчанию заглушает любое воспроизведение звука с веб-страницы (как через `<audio>`/`new Audio()`, так и во многих случаях через Web Audio API) при включённом аппаратном переключателе "Бесшумный режим" (mute switch). В отличие от нативных приложений, у веб-страниц в Safari нет надёжного способа обойти это программно. То есть при включённом mute switch на iPhone звук не сработает вообще, независимо от того, насколько "громко" он замастерен — это ограничение платформы, а не баг кода.
3. `triggerVibration()` (строка 154-162) использует `navigator.vibrate(...)` — этот API **не реализован в Safari/iOS вообще** (ни в мобильном Safari, ни в WebView, ни в PWA на iOS). То есть предложенное в тикете "добавить вибрацию" даст эффект только на Android — на iPhone эта строка тихо ничего не делает (код уже это учитывает через `if ('vibrate' in navigator)`, поэтому не падает, но и не помогает).
4. Визуальная обратная связь (`triggerFlashEffect()`, строки 165-170) уже существует, но: длительность всего 300 мс (тикет просит до 1 секунды) и эффект — статичное зелёное свечение (`box-shadow` через CSS-класс `.flash-success`), не "радужный" градиент, как просит тикет.

**Архитектурный вывод**: на iOS единственный канал обратной связи, который гарантированно сработает у любого пользователя независимо от настроек устройства — **визуальный**. Звук может не сработать (mute switch), вибрация не сработает никогда (нет API). Поэтому усиление звука/вибрации — это "best effort" улучшение для Android и части случаев на iOS, а не решение само по себе; надёжным решением проблемы "непонятно, сработало ли сканирование" является в первую очередь усиленная визуальная индикация, которую и просит тикет.

## Затрагиваемые модули

- `src/views/ScannerView.vue` — единственный компонент, где реализованы выбор камеры (`initScanner`, `startScanning`), звук (`playSound`, `successSound`, `errorSound`), вибрация (`triggerVibration`) и визуальный эффект (`triggerFlashEffect`, CSS `.flash-success`).
- Зависимость: библиотека `html5-qrcode` (`package.json`) — используется её публичный API `Html5Qrcode.getCameras()` и `Html5Qrcode.start()`, поддерживающий запуск как по `deviceId`, так и по constraint-объекту (`{ facingMode: ... }`) — переработка использует уже имеющиеся возможности библиотеки, апгрейд версии не требуется.
- Статические ассеты: текущие base64-звуки встроены прямо в `<script>` — при замене на более качественные звуки стоит решить, оставлять ли data URI или вынести в `public/` (см. "Риски").

## Изменения в существующих модулях

### `src/views/ScannerView.vue` — выбор камеры

- В `initScanner()` заменить эвристику парсинга `device.label` на запуск сканера через семантический constraint `facingMode`:
  - Основной путь: `html5QrCode.start({ facingMode: { exact: "environment" } }, SCANNER_CONFIG, onScanSuccess, onScanError)` — просит у ОС/браузера именно тыловую камеру напрямую, без угадывания по лейблу. Работает единообразно на Android Chrome и iOS Safari.
  - Fallback при ошибке (constraint `exact` может кинуть `OverconstrainedError`, если у устройства физически нет тыловой камеры или браузер не поддерживает `exact`): откатываться на нестрогий `{ facingMode: "environment" }`, а при повторной ошибке — на текущее поведение через `Html5Qrcode.getCameras()` + `devices[0]` (или последний вручную выбранный пользователем `deviceId`, см. ниже), чтобы сканер в любом случае запустился, а не падал в тупик.
  - `Html5Qrcode.getCameras()` при этом не убирается из кода — он остаётся нужен для второго пункта (ручной выбор).

- Добавить **ручной выбор камеры** как явный элемент управления (не только скрытый fallback):
  - Небольшой UI-элемент (иконка/кнопка "сменить камеру" в `scanner-header`, рядом с существующей кнопкой "Beenden") — открывает список камер, полученный через `Html5Qrcode.getCameras()` (id + label).
  - При выборе — остановить текущий сканер (`stopScanning()`), запустить заново с выбранным `deviceId` (`startScanning()` уже поддерживает `CAMERA_ID` как deviceId — переиспользуется).
  - Выбор сохраняется в `localStorage` (ключ, например, `scanner_preferred_camera_id`) и используется как приоритетный вариант при следующем открытии `/scanner` на этом устройстве — до нового автовыбора через `facingMode` (то есть: если пользователь один раз вручную выбрал камеру на этом телефоне, повторно спрашивать/угадывать не нужно).
  - Это закрывает случай нескольких тыловых камер (wide/ultra-wide/telephoto), где `facingMode: environment` может взять не ту, которая физически удобна для сканирования QR на близком расстоянии.

### `src/views/ScannerView.vue` — звук

- Заменить текущие короткие тихие base64-WAV (`successSound`, `errorSound`) на более выразительные сигналы. Два равноценных по сложности варианта:
  - (а) Заменить embedded data URI на новый, более громко/чётко смастеренный короткий звук (более высокая амплитуда, более резкая атака — субъективно "громче" при той же технической громкости 1.0).
  - (б) Сгенерировать сигнал программно через Web Audio API (`AudioContext` + `OscillatorNode`), задав частоту/длительность/огибающую в коде — не зависит от качества исходного файла, легко настраивается (например, короткий двойной "бип" на успех, низкий одиночный тон на ошибку).
  - Рекомендация: (б), т.к. это устраняет зависимость от качества стороннего аудио-ассета и даёт точный контроль громкости/длительности без добавления новых файлов в репозиторий — согласуется с текущим подходом (звук уже был встроен инлайн, а не файлом в `public/`).
- Явно задокументировать в коде (комментарий), что на iOS Safari звук может не воспроизводиться при физическом mute switch — чтобы это не расценивалось как баг при будущем тестировании на iPhone в беззвучном режиме.

### `src/views/ScannerView.vue` — вибрация

- `triggerVibration()` — усилить паттерн для Android (длиннее/сильнее, например `[150, 50, 150, 50, 150]` вместо текущего `[100, 50, 100]`), т.к. на устройствах в кармане/на солнце короткая вибрация тоже может быть незаметна.
- Кода не требуется менять для iOS — проверка `if ('vibrate' in navigator)` уже корректно исключает iOS без ошибок; поведение "на iPhone вибрации не будет" остаётся ожидаемым платформенным ограничением, не багом.

### `src/views/ScannerView.vue` / `<style scoped>` — визуальная индикация

- `triggerFlashEffect()`: увеличить длительность с 300 мс до ~1000 мс (по тикету — "до одной секунды").
- CSS-класс `.flash-success` на `.scanner-container`: заменить статичный зелёный `box-shadow` на анимированную радужную/градиентную рамку (например, CSS `@keyframes` с `background-position`/`border-image` на conic- или linear-gradient, циклически проходящую через цветовой спектр в течение ~1 секунды) с плавным возвратом к исходному виду по завершении анимации.
- Для сценария ошибки (`onScanError`/невалидный QR/ребёнок не найден) — сохранить текущий раздельный визуальный канал (`lastError` алерт уже есть), но стоит рассмотреть отдельный (не радужный, а, например, красный пульсирующий) вариант анимации рамки, чтобы "успех" и "ошибка" визуально не путались — по тикету явно описан только положительный сценарий ("на время до одной секунды"), поэтому для ошибки минимально допустимо оставить как есть (текущий `shake`-эффект алерта), либо симметрично усилить — на усмотрение при реализации, не блокирует основную задачу.

3. Заменить/усилить звуковые сигналы (`successSound`/`errorSound`) — выбрать между новым аудио-ассетом и Web Audio API генерацией (рекомендация — генерация, см. выше), сохранив текущий вызов `playSound()` как точку интеграции.
4. Усилить паттерн вибрации в `triggerVibration()`.

   - iPhone (Safari) — автовыбор тыловой камеры через `facingMode` (ключевая проверка исходного бага), звук при обычном и при включённом mute switch (ожидаемо: в беззвучном режиме звука не будет — проверить, что визуальный эффект при этом всё равно чётко виден), отсутствие ошибок из-за недоступного `navigator.vibrate`.
   - Устройство с несколькими тыловыми камерами — проверить ручной выбор конкретной камеры и его сохранение между сессиями.

## Риски

- **`facingMode: { exact: "environment" }` не гарантированно поддерживается всеми связками браузер/устройство** — обязателен многоуровневый fallback (см. выше), иначе часть пользователей вообще не сможет запустить сканер (регресс хуже текущего состояния, где хотя бы Android работает).
- **Звук на iOS при включённом mute switch не заработает независимо от качества сигнала** — это платформенное ограничение, не решаемое кодом. Нужно явно донести это до заказчика/пользователей (например, короткая подсказка в UI или инструктаж персонала — "проверьте, что телефон не в беззвучном режиме"), чтобы усиленный звук не воспринимался как "опять не работает" в реальном использовании. Визуальный эффект — единственный канал, гарантированно работающий на iOS в любом режиме.
- **`navigator.vibrate` полностью отсутствует в Safari/iOS** — усиление паттерна вибрации даст эффект только на Android; ожидания по тикету ("добавить вибрацию") на iPhone физически нереализуемы веб-технологиями на сегодняшний день (нативное приложение или обёртка типа Capacitor могли бы это обойти, но это выходит за рамки текущей PWA-архитектуры и данного тикета).
- **Ручной выбор камеры, сохранённый в `localStorage`, может "залипнуть" на устройстве**, если, например, к телефону подключили внешнюю камеру один раз и сохранённый `deviceId` перестал существовать — нужна проверка при старте, что сохранённый `deviceId` всё ещё присутствует в текущем списке `getCameras()`, иначе откатываться на автовыбор через `facingMode`.
- Генерация звука через Web Audio API (если выбран этот путь) требует, чтобы `AudioContext` создавался/резюмировался по пользовательскому жесту (браузерные политики автовоспроизведения) — на экране сканера это не проблема, т.к. пользователь уже взаимодействует со страницей (нажатие "начать сканирование"/сама навигация), но стоит явно проверить на iOS Safari, где политики автовоспроизведения особенно строгие.

## Definition of Done

- [ ] На iPhone (Safari) при открытии `/scanner` автоматически включается тыловая (не селфи) камера.
- [ ] На Android (Chrome) автовыбор тыловой камеры продолжает работать как раньше (нет регресса).
- [ ] Пользователь может вручную выбрать конкретную камеру из списка всех доступных камер устройства; выбор сохраняется между визитами на странице.
- [ ] При недоступности предпочтительного constraint/сохранённой камеры сканер всё равно запускается (многоуровневый fallback), а не падает в ошибку.
- [ ] Успешный скан сопровождается заметно более громким/выразительным звуком (на устройствах не в беззвучном режиме), усиленной вибрацией (на Android) и радужным/градиентным flash-эффектом рамки длительностью ~1 секунда с возвратом к исходному виду.
- [ ] На iPhone в режиме "Бесшумный" визуальный эффект по-прежнему чётко сигнализирует об успешном/неуспешном скане, несмотря на отсутствие звука.
- [ ] Отсутствие `navigator.vibrate` на iOS не вызывает ошибок в консоли и не ломает остальную обратную связь.
- [ ] Регрессионно проверена основная функция сканера (распознавание QR, запись присутствия ребёнка, обработка дублей и ошибок) — изменения в этом тикете её не затрагивают.
