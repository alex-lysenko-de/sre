# Итог

CHANGES_REQUIRED

Реализация в целом добротная и близко следует плану: выбор камеры переведён на семантический `facingMode` с каскадным fallback и ручным выбором (подтверждено чтением исходников зависимости `node_modules/html5-qrcode/esm/html5-qrcode.js` — `{ facingMode: { exact: 'environment' } }` действительно официальный, а не угаданный API), визуальный эффект усилен и стал радужным, вибрация усилена. `npx vite build` проходит без ошибок, дефолтных ссылок на старый код (`CAMERA_ID`, старый `startScanning`) не осталось. Однако найдено два содержательных Major-дефекта: (1) уже отмеченный ранее риск, что `AudioContext` резюмируется вне пользовательского жеста и звук рискует не работать вовсе именно на iPhone, и (2) подтверждённая по исходникам библиотеки гонка состояний — параллельный вызов `html5QrCode.start()` при взаимодействии с новым UI выбора камеры во время ещё не завершившейся автоинициализации бросает синхронное исключение внутри библиотеки. Оба требуют конкретных изменений в коде, а не только повторного тестирования.

# Найденные проблемы

## Critical

Нет.

## Major

### 0. Гонка состояний при параллельном вызове `html5QrCode.start()` (новая находка, не покрыта предыдущим проходом ревью)

Кнопка выбора камеры (`scanner-header`, `.btn-camera`, строки 14-20 шаблона) и открывающийся по ней `<select>` доступны сразу после монтирования компонента и ничем не блокируются на время асинхронной инициализации в `initScanner()` (`configStore.loadConfig()` → `Html5Qrcode.getCameras()` → `startScanningWithFacingMode()`, которая может последовательно предпринять до трёх попыток `.start()` с разными constraint'ами, прежде чем сканер реально запустится).

Библиотека `html5-qrcode` использует внутренний `StateManagerImpl` (`node_modules/html5-qrcode/esm/state-manager.js:43-48`), который **синхронно бросает исключение**, если запрашивается переход состояния, пока предыдущий переход ещё не завершён (`execute()`/`cancel()` не вызваны для текущей транзакции):
```js
failIfTransitionOngoing: function () {
    if (this.onGoingTransactionNewState !== Html5QrcodeScannerState.UNKNOWN) {
        throw "Cannot transition to a new state, already under transition";
    }
}
```
Это выполняется в самом начале `Html5Qrcode.prototype.start()` (строка 109 файла), до какого-либо асинхронного `getUserMedia`.

Сценарий: пользователь открывает `/scanner`, почти сразу нажимает кнопку камеры и выбирает пункт в `<select>`, пока фоновая цепочка `startScanningWithFacingMode()` из `initScanner()` ещё не завершилась (типично — от одной до нескольких секунд, особенно если `exact environment` проваливается и происходит вторая попытка). `selectCamera()` (строки 310-324) вызывает `stopScanning()` — no-op, так как `scannerActive.value` в этот момент ещё `false` — и затем `startScanningWithDeviceId(deviceId)` → `html5QrCode.start(...)`, что бросает `"Cannot transition to a new state, already under transition"`. Пользователь видит `alert('Fehler beim Wechseln der Kamera.')`, хотя реальной проблемы с доступом к камере нет — просто два вызова `start()` пересеклись по времени. Параллельно фоновая инициализация продолжает жить своей жизнью независимо от неудавшейся попытки ручного выбора, и итоговое состояние (`currentCameraId`, `scannerActive`, какая физическая камера в итоге активна) становится непредсказуемым и рассинхронизированным с тем, что видит пользователь в UI.

Это прямое следствие того, что новый элемент управления (ручной выбор камеры) не был защищён от пересечения с уже идущим асинхронным запуском — риск, который план не рассматривал явно, но который логически вытекает из добавления интерактивного UI поверх асинхронной инициализации, обёрнутой только в try/catch, а не в семафор/флаг состояния.

**Файл**: `src/views/ScannerView.vue`, кнопка/`<select>` — строки 14-20 и 27-40, `selectCamera` — строки 310-324, `initScanner` — строки 327-352.

**Требуемое исправление**: ввести явный флаг занятости (например, `isTransitioning`) и не давать `selectCamera`/обработчику `<select>` вызывать `startScanningWithDeviceId` пока он установлен, либо отключать (`disabled`) кнопку/`<select>` до первого успешного `scannerActive.value === true`.

### 1. `AudioContext` создаётся и резюмируется вне пользовательского жеста — на iOS Safari звук может не воспроизводиться вообще, независимо от mute switch

### 1. `AudioContext` создаётся и резюмируется вне пользовательского жеста — на iOS Safari звук может не воспроизводиться вообще, независимо от mute switch

`getAudioContext()`/`playTone()` (используются в `playSound()`) вызываются только из `processScannedData()` — то есть в ответ на асинхронное событие распознавания QR-кода библиотекой `html5-qrcode`, а не в обработчике клика/тапа пользователя. `AudioContext` создаётся лениво при первом вызове, и именно этот первый вызов `ctx.resume()` (внутри `playTone()`, строки ~156-160) происходит **не** в трастed user gesture handler.

Планом это было явно предусмотрено как риск ("Генерация звука через Web Audio API... требует, чтобы AudioContext создавался/резюмировался по пользовательскому жесту... на экране сканера это не проблема, т.к. пользователь уже взаимодействует со страницей (нажатие 'начать сканирование'/сама навигация)"). Но при чтении фактического кода это допущение не подтверждается:
- В `ScannerView.vue` нет кнопки/действия "начать сканирование" — `initScanner()` запускается автоматически в `onMounted()` (строки 498-500), без какого-либо клика внутри самого компонента.
- Единственный клик, предшествующий открытию `/scanner`, — это навигационный переход из другого компонента (меню/ссылка). Он происходит в отдельном call stack, задолго до монтирования `ScannerView` и создания `AudioContext`, и по правилам WebKit/Web Audio autoplay policy **не** засчитывается как "жест", разблокирующий именно этот `AudioContext`.

Итог: по правилам Safari/WebKit для Web Audio API (в отличие от чуть более мягкой политики для `<audio>`-элементов, которая использовалась раньше), `AudioContext`, впервые созданный и резюмированный вне прямого обработчика клика/тапа, с высокой вероятностью **останется в состоянии `suspended` навсегда**, и запланированные осцилляторы не издадут звука вообще — то есть замена звукового движка может превратить исходную жалобу "слишком тихо" в "звука нет совсем" именно на iPhone, том самом устройстве, о котором тикет и написан. Это не гипотетическая мелочь: это прямое следствие архитектуры экрана (авто-старт без клика), которое не было учтено при выборе реализации, хотя план прямо предупреждал о необходимости проверить этот момент.

**Требуемое исправление** (не просто "протестировать на устройстве", а изменение кода): обеспечить создание/резюмирование `AudioContext` внутри настоящего пользовательского жеста, например:
- добавить экран/оверлей "Tippen, um zu starten" перед первым вызовом `initScanner()`, и в его обработчике клика/тапа заранее создать и `resume()` `AudioContext` (можно "прогреть" тишиной — запустить и сразу остановить осциллятор с нулевым gain), либо
- слушать первый `touchend`/`click` в любом месте документа на этом экране (например, через `window.addEventListener('touchend', unlockAudio, { once: true })` в `onMounted`) и использовать именно его для первого `resume()`.

Без этого DoD-пункт плана "успешный скан сопровождается заметно более громким/выразительным звуком" не может считаться выполненным на iPhone вне режима "Бесшумный" — а это именно тот случай, где звук в принципе должен быть слышен.

## Minor

### 2. `AudioContext` не закрывается при размонтировании компонента

`onBeforeUnmount()` (строка 502-504) останавливает сканер (`stopScanning()`), но не вызывает `audioContext?.close()`. При многократных заходах на `/scanner` в рамках одной сессии SPA (без полной перезагрузки страницы) каждый новый маунт компонента создаёт новый `AudioContext` в новом закрытии `<script setup>`, а старый явно не освобождается. Браузеры ограничивают количество одновременно "живых" (не закрытых) `AudioContext` (в Chrome, например, где-то в районе нескольких единиц), после чего `new AudioContext()` может начать выбрасывать ошибку. Для типичного использования (один сеанс сканирования подряд) не критично, но стоит закрывать контекст в `onBeforeUnmount`.

### 3. Визуальное состояние `<select>` выбора камеры не всегда отражает реально активную камеру

`currentCameraId` устанавливается в `null` при запуске через `facingMode` (авто-выбор) и в конкретный `deviceId` только при ручном выборе/восстановлении сохранённой камеры. Пока камера работает в режиме авто-выбора (типичный случай при первом посещении страницы), `<select :value="currentCameraId">` получает `null`, что нативно рендерится браузером как выбор первого `<option>` в списке — визуально это может выглядеть так, будто конкретная (первая в списке) камера уже выбрана, хотя по факту активна другая физическая камера, выбранная браузером через `facingMode`. Функционально не ломается (при явном выборе той же по счёту камеры переключение всё равно произойдёт корректно, `deviceId !== null` пройдёт), но вводит в заблуждение визуально. Не блокирует, но стоит по возможности либо не выделять ничего (добавить пустой placeholder-`<option>`, когда `currentCameraId === null`), либо получать реальный `deviceId` активной камеры после старта через `facingMode` (`html5QrCode.getRunningTrackCameraCapabilities()`/аналог в используемой версии библиотеки, если доступно).

### 4. Ручное тестирование на реальных устройствах не проводилось

Разработчик прямо и честно отмечает в `IMPLEMENTATION_REPORT.md`, что проверка выполнена только через `vite build` и чтение кода, без доступа к реальным Android/iPhone устройствам. Учитывая находку Major #1 выше, это тестирование не опционально — именно оно единственное могло бы подтвердить или опровергнуть риск с `AudioContext`. Сам факт отсутствия теста не блокировал бы ревью при отсутствии архитектурных сомнений (как было в тикетах 101/102 для RLS-миграций), но в сочетании с Major #1 — это тестирование должно быть проведено **после** внесения исправления #1, а не вместо него.

# Архитектурные замечания

- Замена парсинга `device.label` на `facingMode` constraint с трёхуровневым fallback (`exact environment` → `environment` → старая эвристика по списку устройств) реализована в точности по плану и выглядит надёжной — старое поведение сохранено как самая последняя линия защиты, а не удалено, что снижает риск регресса для устройств, не поддерживающих constraint вовсе.
- Ручной выбор камеры с сохранением в `localStorage` и проверкой на "протухший" `deviceId` (`savedCameraStillExists`) реализован в точности так, как описано в разделе "Риски" плана — хорошее внимание к деталям.
- Замена `<audio>`/base64-WAV на генерацию тона через Web Audio API соответствует явной рекомендации плана и корректна с точки зрения самого Web Audio API (корректное использование `GainNode` с `exponentialRampToValueAtTime`, корректный расчёт `startTime` для последовательных тонов). Технических ошибок в самой DSP-логике не найдено — проблема (Major #1) не в том, "как" генерируется звук, а в том, "когда" впервые создаётся/резюмируется `AudioContext`.
- Усиление вибрации и увеличение длительности + замена цвета flash-эффекта на радужный градиент — точное, аккуратное соответствие и тикету, и плану; CSS-анимация `rainbow-flash` корректно возвращает `box-shadow` к исходному состоянию на 100% кадре.
- `removeDay`-подобных побочных эффектов на бизнес-логику (`processScannedData`, запись присутствия в БД) нет — изменения полностью локализованы в UI/UX-слое, как и предполагал план ("изменения БД не требуются").

# Рекомендации

1. Устранить гонку состояний (Major #0) — заблокировать/скрыть кнопку и `<select>` выбора камеры до первого успешного старта сканера, либо ввести флаг занятости, исключающий параллельные вызовы `html5QrCode.start()`.
2. Внести исправление для Major #1 — обеспечить создание/резюмирование `AudioContext` внутри настоящего пользовательского жеста, прежде чем полагаться на него как на источник обратной связи.
3. После обоих исправлений — обязательно проверить на реальном iPhone (Safari), в том числе в обычном режиме (не только "Бесшумный", который и так по плану ожидаемо тихий) — что звук действительно слышен при первом же скане после открытия страницы, без необходимости предварительного взаимодействия с интерфейсом, и что быстрое нажатие на кнопку смены камеры сразу после открытия `/scanner` не приводит к ложной ошибке.
4. Добавить `audioContext?.close()` в `onBeforeUnmount()` (Minor #2) — недорогая правка, снижающая риск исчерпания лимита контекстов при многократных заходах на экран в течение сессии.
5. Опционально — устранить визуальную неоднозначность `<select>` при авто-выбранной через `facingMode` камере (Minor #3), например через placeholder-опцию "Automatisch (Rückkamera)".
6. Опционально — различать в терминальном fallback (`startScanningWithDeviceListFallback`) причину "нет доступа к камере" и "камер физически нет", чтобы не показывать вводящее в заблуждение `"Keine Kamera gefunden!"` при отклонённом разрешении.
7. `src/composables/useDays.js` в общем диапазоне коммитов с этим тикетом относится к уже отдельно проверенному тикету 102 (`tickets/102/REVIEW_REPORT.md` → PASS/DONE) — не дефект этого тикета, но стоит впредь не смешивать изменения разных тикетов в одном коммите для более чистого ревью по diff'у.
