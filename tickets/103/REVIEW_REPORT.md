# Найденные проблемы

## Critical

### 1. Ни один пункт плана не реализован

Прочитан текущий `src/views/ScannerView.vue` (последний коммит, трогавший файл — `34f45ca "small fixes"`, датирован задолго до появления тикетов 101-103; рабочее дерево для этого файла чистое, `git diff` пуст). Код идентичен диагнозу из `IMPLEMENTATION_PLAN.md`:

- **Выбор камеры** (`initScanner()`, строки 205-229) — по-прежнему парсит `device.label` (`.includes('back')`, `.includes('rear')`, `.includes('0')`) с фоллбэком на `devices[0]`. Никакого запуска через `facingMode: { exact: "environment" }` нет. Это ровно та эвристика, которую план требовал заменить (root cause бага на iPhone).
- **Ручной выбор камеры** — отсутствует полностью: нет UI-элемента, нет сохранения выбора в `localStorage` (`scanner_preferred_camera_id` из плана нигде не встречается в кодовой базе).
- **Звук** (`playSound`, строки 143-151) — те же встроенные base64-WAV, что описаны в диагнозе плана как "тихие"; `volume = 1.0` не менялся (и не мог "стать ещё громче" — план это и указывал), но требуемая по плану замена/генерация сигнала через Web Audio API не сделана.
- **Вибрация** (`triggerVibration`, строки 154-162) — паттерн `[100, 50, 100]`, план требовал усилить (например, `[150, 50, 150, 50, 150]`). Не изменено.
- **Визуальный эффект** (`triggerFlashEffect`, строки 165-170, и CSS-класс `.flash-success`) — длительность всё ещё `300` мс (план и тикет требуют ~1000 мс), эффект — статичное зелёное свечение (`box-shadow`), не радужный/градиентный, как явно просит и тикет, и план.

Т.е. проблема, описанная в `103.txt` ("на iPhone включается селфи-камера", "слишком тихий звук... непонятно, когда сканирование произошло"), в текущем состоянии кода **никак не устранена** — пользователь на проде продолжит видеть тот же баг.

# Архитектурные замечания

План (`IMPLEMENTATION_PLAN.md`) сам по себе выглядит качественно проработанным и реалистичным:
- Корректно определяет, что `facingMode` constraint — правильное семантическое решение вместо парсинга лейблов, с разумным каскадным fallback (`exact environment` → `environment` → `getCameras()`/`devices[0]`), что важно, т.к. `exact` может кинуть `OverconstrainedError` на части устройств.
- Корректно и честно фиксирует платформенные ограничения iOS (mute switch глушит звук программно необходимо, `navigator.vibrate` отсутствует в Safari/iOS в принципе) как факт, а не как то, что можно "починить кодом" — и справедливо делает вывод, что визуальная индикация должна быть основным, а не дополнительным каналом на iOS.
- Явно предусматривает риск "залипшего" `deviceId` в `localStorage`, если сохранённая камера физически отключена — с требованием проверять её наличие в актуальном списке `getCameras()` при старте.
- DoD в плане прямо привязан к ручному тестированию на реальных устройствах (Android + iPhone, включая режим "Бесшумный"), что уместно, т.к. эмулятор не может проверить камеру/вибрацию/mute switch.

Само по себе наличие такого плана без какой-либо реализации — не архитектурная проблема кода (кода просто нет), но процессная: тикет не должен передаваться на ревью реализации, пока фаза разработки не начата/не завершена.
