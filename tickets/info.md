# Концепция системы учета детей и проведения перекличек

## Назначение системы

Система предназначена для организации поездок детей и проведения
перекличек во время мероприятий.

Основная цель системы --- в любой момент времени знать:

-   сколько детей участвует в мероприятии;
-   где находятся дети;
-   какая группа отвечает за каждого ребенка;
-   все ли дети присутствуют перед каждым перемещением группы.

Система должна максимально упростить работу сопровождающих и сократить
время проведения перекличек.

------------------------------------------------------------------------

# Основные участники

В системе существуют три роли.

### Главный администратор

Отвечает за проведение мероприятия в целом.

Видит все группы, всех детей, все автобусы и общую статистику.

Запускает новые переклички и контролирует их выполнение.

Сейчас В системе предусмотрено 2 главных администратора. Они имеют
одинаковые права.

По крайней мере один администратор должен участвовать в каждой поездке.

------------------------------------------------------------------------

### Руководитель группы (Betreuer)

Каждый руководитель в течение одного дня закрепляется только за одной
группой детей.

На следующий день он может быть назначен уже в другую группу.

Руководитель работает только со своей группой и отвечает за учет детей
этой группы.

------------------------------------------------------------------------

### Ребенок

Каждый ребенок принадлежит только одной группе.

В течение дня группа ребенка не изменяется.

Более того: обычно в течении всего периода группа ребенка не меняется.

В какие - то дни ребенок может отсутствовать например по причине
болезни.

------------------------------------------------------------------------

# Группы

Все дети разделены на группы.

Каждая группа имеет одного или нескольких руководителей.

За безопасность детей отвечает именно руководитель своей группы.

Дети отправляются на автобусах утром и возвращаются вечером. Организация
несет ответственность за каждого ребенка в течение всего времени
проведения мероприятия --- с момента передачи ребенка родителями
сопровождающим перед отправлением автобуса утром и до его передачи
родителям после возвращения автобуса вечером.

В отдельных случаях ребенок может покинуть мероприятие раньше. Например,
если он почувствовал себя плохо, перегрелся на солнце, заболел или
родители заранее запланировали другое мероприятие. Такие ситуации
происходят нечасто, однако система должна учитывать возможность
досрочного выбытия ребенка в любой момент и обеспечивать корректный учет
его присутствия.

------------------------------------------------------------------------

# Рабочий день

Каждый день является самостоятельным мероприятием.

Для каждого дня определяется:

-   список участвующих детей;
-   список руководителей;
-   распределение руководителей по группам;
-   распределение автобусов;
-   все последующие переклички.

На следующий день все назначения могут измениться.

------------------------------------------------------------------------

# Работа без интернета

Система в первую очередь ориентирована на работу онлайн. Полноценный
автономный офлайн-режим с синхронизацией между устройствами не
требуется --- это усложнило бы систему без существенной практической
пользы для описанного ниже рабочего процесса.

Первая регистрация участников (нулевая перекличка) проводится утром во
время посадки в автобус, когда доступ к интернету практически всегда
есть. Именно тогда определяется фактический список детей, участвующих
в мероприятии в этот день, и этот список кэшируется локально на
устройствах:

-   каждый руководитель (Betreuer) хранит локально список своей группы
    и список детей, зарегистрированных на текущий день;
-   главный администратор хранит локально данные по всем группам.

Этого достаточно, чтобы при отсутствии связи проводить любые
последующие переклички (например, после аквапарка, парка аттракционов
или зоопарка) полностью офлайн:

-   просмотр своей группы;
-   сканирование QR-кодов;
-   ручная отметка детей.

При отсутствии связи результат переклички сообщается голосом другим
сопровождающим или главному администратору в виде одного из двух
статусов: «все в порядке» или «есть отсутствующие». Отсутствие
ребенка --- это чрезвычайная ситуация: дальнейшее движение групп
останавливается до полного выяснения, и в этот момент детальный обмен
данными через приложение уже не является критичным.

Не реализуются: синхронизация данных между несколькими устройствами,
обмен данными по Bluetooth и очередь офлайн-событий с последующим
слиянием на сервере.

------------------------------------------------------------------------

# Идентификация ребенка

Каждый ребенок получает браслет с QR-кодом.

Браслет **не является именным**.

Он содержит только уникальный идентификатор.

При необходимости браслет можно заменить на новый.

В этом случае в базе данных изменяется только привязка нового браслета к
ребенку.

Сам ребенок при этом остается тем же самым.

Это позволяет быстро заменять поврежденные или потерянные браслеты без
изменения остальных данных.

------------------------------------------------------------------------

# Использование QR-кодов

QR-код является основным способом идентификации ребенка.

После сканирования система мгновенно определяет:

-   ребенка;
-   его группу;
-   текущий статус участия.

Если использование камеры невозможно, руководитель может отметить
ребенка вручную.

------------------------------------------------------------------------

# Переклички

Перекличка представляет собой отдельное событие.

Главный администратор начинает новую перекличку одной кнопкой.

Все руководители получают уведомление о начале новой переклички.

Во время переклички каждый руководитель подтверждает присутствие детей
своей группы.

Подтверждение может выполняться:

-   сканированием QR-кода;
-   ручной отметкой;
-   любым сочетанием этих способов.

После завершения руководитель подтверждает окончание своей переклички.

------------------------------------------------------------------------

# Первая регистрация дня

Перед началом поездки проводится регистрация участников.

Во время нее определяется, какие дети сегодня участвуют в мероприятии.

В дальнейшем этот список может изменяться.

Например, если ребенок опоздал и приехал позже вместе с родителями.

Все остальные переклички проводятся только среди детей,
зарегистрированных как участники текущего дня.

------------------------------------------------------------------------

# Учет автобусов

Автобусы являются отдельной частью системы.

Во время посадки может использоваться режим быстрого сканирования.

Руководитель группы сканирует всех детей, находящихся в своем автобусе.

Система автоматически определяет фактический состав каждого автобуса.

Если ребенок пересел в другой автобус, его последнее по времени
сканирование считается актуальным.

------------------------------------------------------------------------

# Основные принципы системы

При разработке системы соблюдаются следующие принципы:

-   максимально простые действия для пользователя;
-   минимальное количество кнопок;
-   локальное кэширование данных группы для проведения перекличек без
    постоянного доступа к интернету;
-   быстрое проведение перекличек;
-   возможность ручной работы при отказе камеры;
-   надежность важнее количества функций;
-   вся система должна помогать сопровождающим, а не усложнять их
    работу.

------------------------------------------------------------------------

# Архитектурные принципы

## Текущая архитектура

На текущем этапе система использует **Supabase Cloud** в качестве
серверной платформы. Клиентское приложение взаимодействует с Supabase
напрямую через Supabase SDK. Такая архитектура выбрана для максимально
быстрого запуска первой версии продукта.

## Целевая архитектура

После реализации всей основной функциональности система должна перейти
на архитектуру, в которой клиент не зависит от конкретной базы данных
или способа хранения данных.

Клиентское приложение должно работать только через внутренний API
(слой-адаптер). Доступ к базе данных напрямую из клиентского приложения
считается временным решением, которое будет постепенно заменено.

Архитектура должна позволять использовать различные реализации серверной
части без изменения клиентского приложения.

Поддерживаются два режима работы:

-   **Cloud Mode** --- сервер работает с облачной базой данных (Supabase
    или её замена).
-   **Local Mode** --- сервер работает локально на Android-устройстве
    главного администратора.

Переключение между режимами должно выполняться настройкой адреса API без
изменения клиентского кода.

## Локальный сервер мероприятия

Во время мероприятий может полностью отсутствовать доступ в Интернет и
мобильную сеть.

В этом случае Android-устройство главного администратора выполняет
одновременно несколько функций:

-   включает Wi-Fi Hotspot;
-   предоставляет локальный API;
-   содержит локальную базу данных SQLite;
-   является единственным сервером мероприятия.

Все устройства сопровождающих подключаются к этой точке доступа и
продолжают работать через локальную сеть так же, как при работе через
Интернет.

Таким образом система продолжает работать без внешнего подключения к
сети Интернет, сохраняя единую центральную базу данных и избегая сложной
синхронизации между несколькими устройствами.

## Принцип независимости клиента

Клиентское приложение не должно знать, какая база данных используется на
сервере.

Вся работа с данными осуществляется исключительно через API. Реализация
API может использовать любую подходящую технологию хранения данных
(например, Supabase/PostgreSQL или SQLite), не изменяя бизнес-логику и
пользовательский интерфейс приложения.

## Этапы развития проекта

1.  Завершить реализацию всей основной функциональности на существующей
    облачной архитектуре.
2.  Ввести единый API-слой между клиентом и сервером.
3.  Реализовать локальный сервер для Android с SQLite.
4.  Добавить возможность переключения между облачным и локальным
    режимами работы.
5.  Реализовать буферизацию изменений при кратковременных потерях
    соединения с сервером для повышения устойчивости работы в локальной
    сети.
