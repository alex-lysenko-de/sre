# IMPLEMENTATION_PLAN — Ticket #106: Kopfzählung (Headcount) für "Meine Gruppe"

## Тикет (контекст)

Источник: `tickets/106/106.txt`.

Нужна функция подсчёта/проверки присутствия детей **только своей группы** (группы текущего залогиненного пользователя). Для каждого ребёнка — два независимых индикатора:

1. **Anwesend am Morgen** ("присутствовал утром") — определяется автоматически по факту приезда на автобусе, **read-only**, не меняется в течение дня.
2. **Jetzt anwesend** ("присутствует сейчас") — редактируемый вручную статус, который персонал может менять сколько угодно раз в течение дня во время пересчёта; отсутствие ребёнка должно быть визуально сразу заметно. Изменение сохраняется в БД немедленно, без кнопки "Sichern/Speichern".

Точка входа — кнопка на `/group-edit` и/или `/main`, решение по месту допускается принять на этапе реализации.

## Цель

Дать Betreuer'ам быстрый (1 клик от главного экрана или экрана группы) способ провести пересчёт своей группы в любой момент дня, видя одновременно "приехал ли ребёнок утром" и "здесь ли он сейчас", с мгновенным сохранением изменений и явной визуальной индикацией отсутствующих детей.

## Изменения в существующих модулях

Для headcount это стоит исправить сразу, а не тиражировать тот же паттерн.

## Новые компоненты

### `src/composables/useChildPresence.js`
DB-слой (по конвенции проекта — "store all database queries in a separate module"), изолирует Supabase-запросы к новой таблице `child_group_day`:

- `getTodayDate()` — как в `useScan.js`/`useUser.js` (либо импортировать существующую, не дублировать логику).
- `fetchPresentNowStatus(childIds, date)` — забирает существующие строки `child_group_day` для списка детей и даты одним запросом (`IN`), возвращает `Map<child_id, {is_present_now, updated_at}>`.
- `setPresentNow(childId, isPresent, date, updatedByUserId)` — **upsert по паттерну, уже принятому в проекте** (`useUser.js: upsertScheduleField` — сначала `SELECT` на существование записи по `(child_id, day)`, затем `UPDATE` или `INSERT`; либо нативный `supabase.upsert()` с `onConflict: 'child_id,day'`, что эквивалентно, но короче — архитектурно предпочтительнее, т.к. исключает гонку между `SELECT` и `INSERT` при параллельных вызовах).

Комбинирование "утро" (из `useScan.getPresentChildrenForDate`) и "сейчас" (из этого модуля) выполняется на уровне View, а не глубоко внутри composable — держим composables узкими и независимыми, как и остальные в проекте (`useScan`, `useChildren`, `useSupabaseUser` не знают друг о друге).

Pinia-стор для этой фичи **не создаётся**: данные пересчёта нужны только на одном экране и не используются другими компонентами/маршрутами (в отличие от `userInfo`/`config`, которые нужны глобально) — заводить стор было бы избыточной абстракцией, что противоречит принципу "Minimalism & Clarity" из `readme.md`. Состояние остаётся локальным в `HeadcountView.vue`, как это уже сделано в `GroupEditView.vue` и `ChildDetailView.vue`.

### `src/views/HeadcountView.vue`
Новый маршрутизируемый экран (`/headcount`):
- Заголовок: "Kopfzählung — Gruppe {{ group_id }}".
- Список детей `userStore.userInfo.group_id` (через `fetchChildrenByGroup`).
- На каждую строку — два индикатора:
  - **Anwesend am Morgen**: бейдж/иконка (✅/❌), опционально номер автобуса, если есть (`bus_id` из `getPresentChildrenForDate`) — read-only, без обработчика клика.
  - **Jetzt anwesend**: переключатель (Bootstrap `form-check form-switch` или toggle-кнопка) — при `@change` сразу вызывает `useChildPresence().setPresentNow(...)`, локально обновляет состояние ряда (optimistic update) и показывает spinner/ошибку при сбое запроса, аналогично `ChildPresenceModal.vue` (`loading`/`error` через `ref`).
- Явная визуальная индикация отсутствия: строка с `Jetzt anwesend = false` получает `bg-danger-subtle`/красную полосу слева + текст "❌ Fehlt", по аналогии с уже используемыми `alert-warning`/`alert-danger` стилями в `ChildDetailView.vue`.
- Без кнопки "Speichern" — по требованию тикета каждое изменение сохраняется сразу.
- Кнопка "Zurück" → `/main` (как в `GroupEditView.vue`, `SelectChildView.vue`).

## Изменения БД

**Новая таблица `child_group_day`** — по образцу уже существующей `user_group_day` (та же модель "статус на конкретный день, upsert по ключу"), которая используется для аналогичной задачи, но для персонала (`isPresentToday` в `user_group_day`, см. `useSupabaseUser.js:78-170`, `stores/user.js:185-211`). Переиспользование уже проверенного в проекте паттерна вместо изобретения нового.

```
child_group_day
├─ id              bigint PK
├─ child_id        bigint  NOT NULL  FK → children(id)
├─ day             date    NOT NULL
├─ is_present_now  boolean NOT NULL DEFAULT true
├─ updated_by      bigint  NULL      FK → users(id)   -- кто последний раз отметил (аудит)
├─ updated_at      timestamptz NOT NULL DEFAULT now()
└─ UNIQUE (child_id, day)
```

- "Утренний" статус **не хранится отдельно** — он уже полностью выводим из существующей таблицы `scans` (`type=1` "Präsenz" и/или наличие `bus_id`), никаких изменений схемы `scans`/`children` не требуется.
- Значение по умолчанию при первом открытии экрана в течение дня (пока строки `child_group_day` для ребёнка ещё нет): показываем `Jetzt anwesend = Anwesend am Morgen` (то есть если ребёнок утром приехал — по умолчанию считаем его присутствующим, пока Betreuer явно не отметит обратное). Строка в `child_group_day` создаётся только в момент первого реального переключения, не заранее (без batch-seed job на начало дня).

**Важно (риск, см. ниже):** в репозитории нет версионируемых SQL-миграций (`supabase/` содержит только Edge Functions, схема БД управляется вручную через Supabase Dashboard) — создание таблицы придётся выполнить вручную и задокументировать отдельно (например, SQL-скрипт в тикете/PR-описании), т.к. воспроизвести схему через `git` невозможно.

## Риски

1. **Отсутствие RLS-разграничения по группам — существующий архитектурный пробел, который эта фича унаследует.** Сейчас клиент сам фильтрует данные по группе (`GroupEditView.vue`, computed `children`), а не БД — то есть теоретически через прямой вызов Supabase API обычный пользователь может прочитать/изменить данные чужой группы. Без RLS-политики на `child_group_day` то же самое будет верно и для статусов присутствия. Это не новый риск, но эта фича — первый случай, когда "запись" (не только чтение) обычного пользователя должна быть строго ограничена своей группой, поэтому стоит закрыть его именно сейчас, а не откладывать.
2. **Нет версионируемой схемы БД в репозитории.** Создание новой таблицы вручную через Dashboard → риск расхождения между dev/prod окружениями, невозможность отследить историю схемы через git (аналогичная ситуация уже существует для всех остальных таблиц проекта, не блокер, но стоит зафиксировать).
3. **Открытый вопрос: должен ли переключатель "Jetzt anwesend" быть заблокирован для детей с `Anwesend am Morgen = false`?** Решение в этом плане — **не блокировать**: Betreuer должен иметь возможность вручную отметить ребёнка присутствующим даже если утренний скан не сработал (разряженный браслет, забыли отсканировать, ребёнка привезли не на автобусе) — иначе headcount не сможет исправить ошибку автоматического утреннего трекинга, а это прямо противоречит цели функции (безопасность детей). Если продукт хочет иной вариант (жёсткая блокировка) — нужно явное решение заказчика до реализации.
4. **Дети без `band_id`** (поле nullable) никогда не получат "утренний" scan через механизм браслета → `Anwesend am Morgen` всегда будет `false` для них, даже если ребёнок физически приехал. Не баг новой фичи, а следствие существующей архитектуры сканирования, но при headcount это будет заметно (весь блок "утро" всегда красный) — стоит явно уточнить у заказчика, ожидаемо ли это.
5. **Множественные Betreuer одной группы одновременно.** Без realtime-подписки (см. п.10 выше) два сотрудника, считающих одну группу параллельно на разных телефонах, не увидят изменений друг друга без ручного обновления страницы — рассинхронизация. Не в скоупе тикета, но стоит явно зафиксировать как известное ограничение первой версии.
6. **iOS-специфика.** Проект уже испытывает проблемы с локальным хранилищем на iOS (см. тикет 105) — сам headcount не хранит ничего локально (все изменения идут напрямую в БД), поэтому напрямую этот риск не наследуется, но стоит протестировать именно на iPhone из-за прежних прецедентов нестабильного поведения PWA на этой платформе.
