# Итог

CHANGES_REQUIRED

Реализация корректно следует ревизии 2 плана: переиспользует `children_today`/`groups_today` вместо параллельной таблицы, вводит снэпшот `presence_morning`, ограничивает список групой пользователя, даёт мгновенный toggle с optimistic update и явной красной индикацией отсутствия, добавляет SQL-скрипт с RLS. `npm run build` проходит (316 модулей, без ошибок). Но найден один Critical баг в самом ядре фичи (тихий no-op записи при отказе RLS) и коммит содержит существенный посторонний шум, мешающий ревью и истории изменений.

# Найденные проблемы

## Critical

**C1. `setPresentNow` может «успешно» ничего не записать в БД — ветка UPDATE не проверяет число задетых строк.**
Файл: `src/composables/useChildPresence.js:53-62`.

```js
const { error } = await supabase
    .from('children_today')
    .update({ presence_now : presenceNow, user_id : currentUserId })
    .eq('child_id', childId);
```

Postgres RLS `USING`-политика делает недоступные строки просто невидимыми для `UPDATE`: если политика не пропускает строку (пользователь не найден в `user_group_day` на сегодня, `isPresentToday` ещё не подтверждён, миграция `doc/db/headcount_presence_morning.sql` ещё не применена в Supabase, RLS вообще не даёт `UPDATE`-доступа и т.д.), `UPDATE` обновит 0 строк и Supabase/PostgREST вернёт **успех** (`error: null`), а не ошибку. `HeadcountView.vue:141-145` в этом случае не откатывает переключатель и не показывает `child.error` — пользователь видит "успешно" переключённый статус, при этом в БД (и, соответственно, в `groups_today`/admin-панели/у других Betreuer той же группы) ничего не изменилось.

**Сценарий отказа:** ровно тот, что уже описан в `IMPLEMENTATION_REPORT.md` как открытый пункт — фронтенд задеплоен, а `doc/db/headcount_presence_morning.sql` ещё не выполнен вручную в Supabase Dashboard (для всех детей, у кого уже есть строка в `children_today` — то есть подавляющее большинство после утренних сканов). Betreuer щёлкает "Jetzt anwesend" → OFF, экран показывает красную строку "Fehlt", но реальная БД (и то, что видит админ) остаётся без изменений — при пересчёте по чек-листу заказчика ("проверка, что не создался параллельный источник правды", DoD плана) это не будет замечено, т.к. UI лжёт об успехе. То же самое может произойти и после применения миграции — при любом расхождении между `user_group_day` пользователя и `children_today.group_id` (например, пользователь ещё не подтвердил "isPresentToday" в своём расписании на сегодня).

Ветка `INSERT` (для ещё не сканированных детей) не имеет этой проблемы — там `WITH CHECK` при отказе действительно возвращает ошибку 42501, которая корректно попадает в `catch`.

**Исправление:** добавить `.select().single()` (или `.select()` + проверка `data.length`) к `UPDATE`-запросу и трактовать пустой результат как ошибку — по аналогии с уже существующим в проекте паттерном `bindBraceletToExistingChild` (`useChildren.js:50-55`, тоже `.update().select().single()`), который для этой же проблемы решение уже применяет.

## Major

**M1. Коммит смешивает фичу с посторонним переформатированием (CRLF) трёх файлов, что скрывает реальный diff.**
`git show --stat HEAD` показывает `GroupEditView.vue` (604 строки изменено), `MainView.vue` (672), `main.js` (246), `db_triggers.sql` (569) — притом что фактические функциональные изменения в каждом из них укладываются в 5-20 строк (проверено через `git diff -w`). Причина — конвертация LF → CRLF всего файла, никак не связанная с тикетом 106. Это затрудняет ревью (реальные изменения тонут в шуме), ломает `git blame` для всех строк файла и создаёт риск конфликтов при параллельной работе других разработчиков с этими же файлами. Рекомендация: перед коммитом настроить `.gitattributes`/редактор так, чтобы файлы сохраняли исходные окончания строк, и в этом PR закоммитить только реальные изменения (можно переприменить патч поверх исходных окончаний строк).

**M2. В тот же коммит попал не относящийся к тикету `rename {tickets => doc}/info.md`.**
Перемещение файла концепции не упомянуто ни в `IMPLEMENTATION_PLAN.md`, ни в `IMPLEMENTATION_REPORT.md` — неясно, преднамеренно ли оно и почему сделано именно в рамках 106. Если это не связано с тикетом, стоит вынести в отдельный коммит/PR — сейчас это затрудняет привязку истории git к конкретной задаче.

**M3. Гонка при параллельной вставке одним и тем же ребёнком (риск №2 плана) реализована не полностью — retry/catch на unique violation отсутствует.**
`IMPLEMENTATION_PLAN.md` (риск №2) явно предполагал обработку конфликта уникальности при параллельной вставке через `try/catch`, `IMPLEMENTATION_REPORT.md` заявляет только "раздельные SELECT→UPDATE/INSERT", но код (`useChildPresence.js:63-72`) не перехватывает ошибку unique constraint (`children_today_child_id_key`) при `INSERT`. Если два Betreuer одной группы одновременно впервые отмечают одного и того же несканированного ребёнка (оба видят "строки нет" в `SELECT`), один `INSERT` получит raw-ошибку Postgres (`23505`), которая всплывёт как `child.error` с непонятным для персонала текстом вместо тихого фоллбэка на `UPDATE`. Не блокирует релиз (это безопасный отказ, не потеря данных), но стоит либо реализовать фоллбэк, как задумано планом, либо явно зафиксировать отклонение от плана в отчёте (сейчас `IMPLEMENTATION_REPORT.md` не упоминает, что эта часть плана не реализована).

## Minor

**N1. RLS для `SELECT` на `children_today` остаётся `USING (true)` — межгрупповое чтение presence-данных технически не ограничено.**
Новый скрипт `doc/db/headcount_presence_morning.sql` добавляет `UPDATE`/`INSERT`-политики, но не трогает существующую политику `authenticated_read_children_today` (`FOR SELECT USING (true)`, `doc/db/adminBusView.md:446-449`). Фильтрация по группе в `getTodayGroupPresence` — только на клиенте; технически любой авторизованный пользователь может запросить `children_today` напрямую через Supabase JS SDK и увидеть presence-данные чужих групп. Это предсуществующий пробел проекта (не введён этим тикетом), но тикет 106 формулирует "the feature only applies to children who belong to the user's own group" — стоит хотя бы зафиксировать как известное ограничение, раз уж write-сторона в этом же PR получила защиту.

**N2. Побочный эффект на агрегаты admin-панели: ручная отметка ребёнка, ни разу не сканированного, засчитывается в `groups_today.children_today` наравне со сканированными.**
При `INSERT` в `setPresentNow` (ветка "строки ещё нет") явно не указывается `presence_today`, поэтому используется DB-дефолт `1` (осознанно, см. план 0.1). Триггер `on_children_today_change` считает `children_today` (агрегат "был сегодня") по `presence_today = 1` без различия источника. В результате admin-панель (`AdminBusView.vue`) после ручной пометки ребёнка, который физически не сканировался и не приезжал на автобусе, покажет его в счётчике "детей сегодня" наравне с реально прибывшими. Решение плана 0.1 сознательно это допускает, но эффект стоит явно проверить на реальных данных при ручном тестировании DoD — сейчас это не входит ни в один пункт чек-листа "Ручные проверки" `IMPLEMENTATION_REPORT.md`.

**N3. `HeadcountView.vue` не проверяет отсутствие `groupId` перед запросом.**
`created()` присваивает `this.groupId = this.userStore.userInfo.group_id` без проверки на `null`/`undefined` и сразу вызывает `loadInitialData()`. Кнопки входа (`MainView.vue`, `GroupEditView.vue`) корректно скрыты при отсутствии группы, но прямой переход по URL `/headcount` (маршрут не имеет `requiresGroup`-подобного guard'а, только `requiresAuth`) у пользователя без группы на сегодня приведёт к `fetchChildrenByGroup(null)`/`getTodayGroupPresence(null)` — по повдению Supabase `.eq('group_id', null)` вероятен пустой список без внятного сообщения, а не понятная ошибка "у вас нет группы на сегодня". Низкий приоритет — тот же класс проблемы уже существует у `/group-edit` без параметра.

# Архитектурные замечания

- Ключевое архитектурное решение ревью плана (Critical #1 — переиспользовать `children_today`/`groups_today` вместо новой таблицы) реализовано верно и полностью: отдельного источника правды не создано, `on_children_today_change` продолжает питать `groups_today`/admin-панель для записей, сделанных через Headcount, без дополнительной интеграции — проверено чтением `doc/db_triggers.sql`.
- Семантика "Anwesend am Morgen" (Major #3 ревью плана) закрыта корректно: новое поле `presence_morning` пишется только в INSERT-ветке `on_scan_insert` и никогда не переписывается — реализация в `doc/db/headcount_presence_morning.sql` и `doc/db_triggers.sql` идентична и синхронна между собой.
- UX-баг с кнопкой на `GroupEditView.vue` для админов, просматривающих чужую группу (Minor #4 ревью плана), исправлен именно так, как предлагал план (`canShowHeadcountButton`), diff подтверждён.
- Основной новый архитектурный риск — не в структуре данных (она выбрана верно), а в надёжности клиентского слоя поверх RLS (см. Critical C1): проект нигде системно не проверяет "0 affected rows = возможно, RLS-отказ", это стоит рассмотреть как общий паттерн для будущих pull-запросов, пишущих через RLS-ограниченные таблицы, а не только для этого экрана.
