Нужно создать полноценную базу знаний проекта в формате **Obsidian** (vault с markdown-файлами и внутренними wiki-ссылками `[[...]]`).

Это не должен быть набор разрозненных заметок — нужна единая, взаимосвязанная документация, описывающая систему целиком: как она устроена, почему она устроена именно так, и как её части соотносятся друг с другом.

## Что должно быть отражено

* **Архитектура проекта** — Vue 3 (Composition API), Pinia, Supabase (Postgres + Edge Functions), структура каталогов `src/` (components, composables, modules, views, router, stores), текущая архитектура vs. целевая (см. раздел "Архитектурные принципы" в `doc/info.md`: переход на API-слой, Cloud Mode / Local Mode).
* **Основные принципы системы** — предметная область (переклички, группы, автобусы, Betreuer), роли пользователей, рабочий день как отдельное мероприятие (см. `doc/info.md`).
* **Структура базы данных** — все таблицы Supabase (`children`, `scans`, `days`, `users`, `user_group_day`, `config`, `groups`, `buses` и т.д.), их поля, связи, RLS-политики, триггеры. Актуализировать и объединить то, что сейчас разрозненно лежит в `doc/table_structure.md`, `doc/db/`, `doc/db_triggers.sql`, `doc/database_migration_config_rls.sql`.
* **Модель учета пользователей** — аутентификация через Supabase Auth, роли (главный администратор / Betreuer), связка `useUser.js` / `useSupabaseUser.js` / `stores/user.js`, кэширование через LocalForage (см. `doc/useUser.md`).
* **API / composables** — обзор всех `src/composables/useXXX.js` как фактического API-слоя приложения (какие данные и операции предоставляет каждый).
* **Используемые URL и маршруты** — карта `src/router/index.js`, включая публичные QR-код URL (см. `doc/QR-codes.md`) и админ-маршруты.
* **Прочие технические аспекты** — офлайн-поведение (см. обновлённый раздел "Работа без интернета" в `doc/info.md`), сборка/деплой (GitHub Pages, GitHub Actions), инструменты `project_packer.py` / `pack.bat` / `unpack.bat`.

## Требования к структуре

* Каждая тема — отдельная заметка (файл).
* Заметки должны ссылаться друг на друга через `[[wiki-ссылки]]`, чтобы по документации было удобно перемещаться (например, заметка про таблицу `scans` должна ссылаться на заметку про composable `useArmband.js`, который её использует, и наоборот).
* Нужна заметка-индекс (MOC — map of content), с которой начинается навигация по всей базе знаний.
* Где уместно — объяснять не только "как", но и "почему" (причины архитектурных решений), опираясь на существующие материалы в `doc/` (после ревизии в тикете 107) и на исходный код.

## Источники

За основу взять актуализированные документы из `doc/` (после ревизии, проведённой в тикете 107 — см. `tickets/107/AUDIT_REPORT.md`) и, где документации недостаточно или она устарела, — реверс-инжиниринг текущего исходного кода в `src/`.

## Результат

Эта база знаний должна стать единым источником правды о проекте и основой для:

* пользовательской документации (тикет 109);
* инструкций для сопровождающих;
* объяснения принятых архитектурных решений новым участникам команды.
