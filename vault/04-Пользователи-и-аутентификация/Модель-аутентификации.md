# Модель аутентификации

> Источники: `doc/useUser.md` (обновлён в тикете 107 под реальную
> 3-слойную архитектуру), `src/router/index.js`.

Аутентификация — через **Supabase Auth**. Приложение поддерживает две
прикладные роли поверх `auth.users`: `admin` и `user` (Betreuer), хранимые в
таблице [[users]] (см. [[Роли-пользователей]]).

## Вход в систему выполняется один раз

По задумке (`doc/selectGroupAndBus.md`), `LoginView.vue` пользователь видит
только один раз — при первой регистрации. Дальше сессия сохраняется, и вход
выполняется автоматически («в тихом режиме»). Токен сессии хранится через
адаптер `supabaseStorageAdapter` поверх LocalForage (см.
[[Кэширование-LocalForage]]), а не в обычном `localStorage`.

## Реализация в три слоя

Изначально задание (`doc/useUser.md`) предполагало один модуль
`src/modules/useUser.js`. По факту реализовано как три отдельных слоя:

```
Components (UI)
     ↓
useUser.js       — бизнес-логика: кэш (LocalForage, TTL), оркестрация
     ↓
useSupabaseUser.js — только запросы к Supabase (getSession, fetchUserFromDB,
                      fetchScheduleForDate, updateScheduleRecord, signOut, ...)
     ↓
stores/user.js   — Pinia store: только реактивное состояние (userInfo) +
                    геттеры (isAdmin, isCheckInRequired, userEmail); методы
                    делегируют в useUser()
```

Подробности каждого слоя — [[useUser]], [[useSupabaseUser]], [[stores-user]].

## Navigation guard (`src/router/index.js`)

`router.beforeEach` выполняет несколько проверок последовательно:

1. Гостевой доступ — если пользователь ни разу не регистрировался
   (`sre_user_registered` отсутствует в LocalForage), разрешены только
   публичные страницы, `Welcome`, `Main`, `Info`.
2. Для маршрутов `meta.public` — пропуск без проверок.
3. Для `meta.requiresAuth` — проверка активной сессии Supabase, затем
   `role`/`active` из [[users]]; неактивный аккаунт — принудительный
   `signOut()` и редирект на `/login`.
4. Для `meta.requiresAdmin` — доп. проверка `role === 'admin'`, иначе
   редирект на `/main` с алертом.
5. Обновление `users.last_seen_date` (fire-and-forget).
6. Перехват `?id=`/`?n=` в query-параметрах любого маршрута → редирект на
   `/armband/:id` (см. [[QR-код-URL]]).

Это дублирует часть логики RLS на клиенте (UX — быстрый редирект без
обращения к серверу), но не заменяет её — реальное ограничение доступа к
данным обеспечивается [[RLS-политики]] на стороне БД.
