# Работа без интернета

> Источник: `doc/info.md` (актуализированный раздел).

Система в первую очередь ориентирована на работу онлайн. Полноценный
автономный офлайн-режим с синхронизацией между устройствами **не требуется**
— это усложнило бы систему без существенной практической пользы для
описанного ниже рабочего процесса.

## Схема

Первая регистрация участников (нулевая перекличка, см.
[[Первая-регистрация-дня]]) проводится утром во время посадки в автобус —
когда доступ к интернету практически всегда есть. Именно тогда определяется
фактический список детей на день, и он кэшируется локально на устройствах:

- каждый Betreuer хранит локально список своей группы и список детей,
  зарегистрированных на текущий день;
- главный администратор хранит локально данные по всем группам.

Этого достаточно, чтобы при отсутствии связи проводить любые последующие
переклички (после аквапарка, парка аттракционов, зоопарка) полностью
офлайн: просмотр своей группы, сканирование QR-кодов, ручная отметка детей.

При отсутствии связи результат переклички сообщается голосом («всё в
порядке» / «есть отсутствующие») — отсутствие ребёнка является чрезвычайной
ситуацией, движение групп останавливается до полного выяснения, и в этот
момент детальный обмен данными через приложение уже не критичен.

## Что осознанно НЕ реализуется

- синхронизация данных между несколькими устройствами;
- обмен данными по Bluetooth;
- очередь офлайн-событий с последующим слиянием на сервере.

## Соответствие в коде

В текущей реализации локальное кэширование через LocalForage покрывает
пользовательские данные (`user_info_cache`, TTL 1 минута, см.
[[Кэширование-LocalForage]]) — это ближе к ускорению повторной загрузки, чем
к полноценному офлайн-хранилищу списков групп/детей, описанному здесь. Полное
покрытие сценария «просмотр группы и сканирование полностью без сети»
локальным кэшем данных детей/группы — не подтверждено в коде на момент
ревизии (тикет 108); это фиксируется как расхождение между описанием
бизнес-процесса и текущей реализацией, а не как ошибка документации.

## Почему

Полная офлайн-синхронизация между устройствами радикально усложнила бы
модель данных (конфликты, слияние, очереди событий) ради сценария,
который на практике решается голосовым сообщением о статусе «все
присутствуют / есть отсутствующие» — что уже достаточно для остановки
движения группы в чрезвычайной ситуации.
