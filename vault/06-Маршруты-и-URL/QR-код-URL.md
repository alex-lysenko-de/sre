# QR-код URL

> Источник: `doc/QR-codes.md` (оставлен как есть в тикете 107 — соответствует
> роутеру и `doc/utils/qr-gen1.html`).

## Текущий формат (реализовано)

```
https://sre-nrw.de?id={id}
```

Например: `https://sre-nrw.de?id=1`. Роутер перехватывает `id` (а также
устаревший алиас `n`) как query-параметр **любого** маршрута и
перенаправляет на `/armband/:id` (см. [[Карта-маршрутов]], `router.beforeEach`
в конце файла).

Генерация самих QR-кодов для печати на браслетах — `doc/utils/qr-gen1.html`
(настраиваемое имя параметра, масштабирование узора; более старый
`qr-gen.html` с фиксированным `n=` помечен в тикете 107 как «УДАЛИТЬ
(рекомендуется)»).

## Предложенный будущий формат (не реализовано)

`doc/QR-codes.md` описывает архитектуру с явным tenant-идентификатором —
цитируется здесь как задокументированное **предложение**, не текущее
поведение:

```
https://sre-nrw.de/{tenant}/{id}
```

где `tenant` — ровно две латинские строчные буквы (например `id`, `aa`,
`bb`), дающие ёмкость 676 независимых пространств ID автобусных номеров.
После логина сервер хранит `tenant` в сессии/JWT; внутренние маршруты
(`/edit/101`, `/history/101`) tenant не содержат — он берётся из сессии.
Публичный QR-URL — единственное место, где tenant должен присутствовать
явно; при сканировании сервер сверяет tenant из URL с tenant сессии и либо
даёт доступ, либо показывает ошибку «браслет принадлежит другой
организации».

## Почему это предложение существует

Текущий формат `?id={id}` — глобальное пространство идентификаторов
браслетов на всё приложение. Формат с tenant предназначен для
мультитенантности (несколько независимых организаций/лагерей на одном
инстансе приложения) — актуально только если проект перейдёт к
обслуживанию нескольких заказчиков одновременно; на момент ревизии (тикет
108) в коде признаков мультитенантности нет.
