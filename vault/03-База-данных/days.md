# `days`

> Источник: `doc/table_structure.md`, `doc/db/days_rls.sql` (тикет 102).

Список дней мероприятия (расписание) — независим от [[reset_events]]/
[[children_today]], которые описывают *состояние* конкретного дня, а не сам
факт его существования в расписании.

```sql
create table public.days (
  id bigint generated by default as identity not null,
  created_at timestamptz not null default now(),
  date character varying not null unique,
  name character varying null,
  abfahrt time null,
  ankommen time null,
  description text null,
  constraint days_pkey primary key (id)
);
```

- `abfahrt`/`ankommen` — время отправления/прибытия автобуса в этот день.
- Управляется через `DaysEditView.vue` (`/days-edit`, admin-only) и
  [[useDays]].`fetchDaysList()`/`saveDay()`/`deleteDay()`.

## RLS (тикет 102)

`SELECT` — любой аутентифицированный пользователь (нужно для `/days-edit` и
других экранов, показывающих расписание). `INSERT`/`UPDATE`/`DELETE` —
только активные `admin`. См. [[RLS-политики]].

### Почему потребовалась отдельная миграция

До этой миграции RLS была включена на `public.days`, но политики `DELETE` не
существовало. PostgREST не считает «0 строк удалено политикой RLS» ошибкой —
поэтому `deleteDay()` в [[useDays]] всегда возвращал успех, даже если
физически ничего не удалялось. `useDays.deleteDay()` теперь явно проверяет
`data.length === 0` после `DELETE ... RETURNING` и выбрасывает ошибку — это
клиентская подстраховка поверх серверного фикса RLS.
