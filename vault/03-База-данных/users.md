# `users`

> Источник: `doc/table_structure.md` (таблица добавлена в ревизии тикета 107 —
> отсутствовала в исходном `table_structure.md`, хотя активно используется в
> коде). RLS-политики и бутстрап SuperAdmin находятся в `doc/users.sql` — этот
> файл **содержит реальные учётные данные администратора** и намеренно не
> цитируется здесь (см. `_Конвенции.md`, пункт 6, и
> `tickets/107/AUDIT_REPORT.md`).

Таблица пользователей приложения — обёртка над `auth.users` Supabase Auth
с прикладными полями (роль, отображаемое имя, активность).

```sql
create table public.users (
  id bigint generated by default as identity not null,
  user_id uuid references auth.users(id) on delete cascade,
  email text not null unique,
  phone text null,
  display_name text not null,
  role text not null check (role in ('admin', 'user')),
  active boolean default true,
  last_seen_date timestamptz null,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  constraint users_pkey primary key (id)
);
```

- `id` (bigint) — внутренний ID, используется во всех прикладных FK
  ([[user_group_day]].`user_id`, [[groups_today]].`user_id` и т.д.) — **не
  путать** с `user_id` (uuid), который ссылается на `auth.users`.
- `role` — `'admin'` или `'user'` (Betreuer), см. [[Роли-пользователей]].
- `active` — деактивированные пользователи не проходят navigation guard (см.
  [[Карта-маршрутов]]).
- `last_seen_date` — обновляется при каждой успешной проверке маршрута
  (`router.beforeEach`).
