# Триггеры

> Источник: `doc/db_triggers.sql` (единственный источник истины —
> `doc/db/triggers.md` был побайтово идентичным дубликатом, помечен в тикете
> 107 как «УДАЛИТЬ (рекомендуется)»). Точечные обновления —
> `doc/db/headcount_presence_morning.sql` (тикет 106).

Вся логика пересчёта агрегированных счётчиков присутствия находится **в
БД** (PL/pgSQL-функции + триггеры), а не в клиентском коде — клиент только
вставляет строки в `scans`/`reset_events`, остальное происходит
автоматически.

## `on_scan_insert` (AFTER INSERT ON `scans`)

Находит `group_id` ребёнка, затем `INSERT ... ON CONFLICT (child_id) DO
UPDATE` в [[children_today]]:

- INSERT-ветка (первый скан ребёнка за всё время существования строки):
  `presence_today=1`, `presence_now=1`, `bus_today=NEW.bus_id`,
  `bus_now=NEW.bus_id`, `presence_morning = CASE WHEN NEW.bus_id IS NOT NULL
  THEN 1 ELSE 0 END` (тикет 106).
- `ON CONFLICT DO UPDATE`-ветка (повторный скан того же дня): только
  `presence_now=1`, `bus_now=EXCLUDED.bus_now`, `user_id=EXCLUDED.user_id`.
  **`presence_morning` сюда сознательно не входит** — это разовый снимок,
  который не должен «мигать» при повторных сканах (см. [[children_today]]).

## `on_children_today_change` (AFTER INSERT OR UPDATE ON `children_today`)

Пересчитывает [[groups_today]] для затронутой группы: `COUNT(*) FILTER
(WHERE presence_today = 1)` и `... presence_now = 1`, затем upsert в
`groups_today`. Полный пересчёт по группе, а не инкремент/декремент — проще
и устойчивее к гонкам, но дороже при больших группах (не проблема при
масштабах этого проекта).

## `on_reset_event_insert` (AFTER INSERT ON `reset_events`)

`CASE NEW.event_type`:

- `0` (Total reset) — `DELETE FROM children_today; DELETE FROM groups_today;`
- `1` (Normal reset) — если это первый reset дня (`event_type=1`) —
  `groups_today.children_today = children_now`, затем `children_now = 0`;
  иначе только `children_now = 0`. В обоих случаях —
  `children_today.presence_now = 0`.
- `2` (Soft reset) — `presence_now`/`children_now` обнуляются без сохранения
  снимка.

См. [[reset_events]] для таблицы событий.

## `on_children_today_delete` (AFTER DELETE ON `children_today`, опционально)

Пересчитывает или удаляет строку [[groups_today]] для группы удалённой
записи, если явное удаление строк `children_today` происходит вне общего
`event_type=0`.

## `recalculate_groups_today()` — вспомогательная функция

Полный пересчёт [[groups_today]] по всем группам сразу — на случай
рассинхронизации счётчиков; вызывается вручную, не привязана к триггеру.

## Индексы

`idx_children_today_group_presence(group_id, presence_today, presence_now)`,
`idx_reset_events_day_type(day, event_type)`, `idx_scans_child_date(child_id,
date)`, `idx_scans_band_date(band_id, date)` — под характерные запросы
подсчёта присутствия и истории сканов.
