# Сборка и деплой

## Стек сборки

- **Vite** — dev-сервер (`npm run dev`) и продакшн-сборка (`npm run build`)
- **npm** (Node.js 20+) — пакетный менеджер

## Деплой

Автоматический через **GitHub Actions**: push в ветку `main` →
деплой на **GitHub Pages** (`.github/workflows/deploy.yml`).

## Вспомогательные инструменты (корень репозитория)

| Инструмент | Назначение | Статус (см. `tickets/107/AUDIT_REPORT.md`) |
|---|---|---|
| `pack.bat` / `unpack.bat` | Вызывают `project_packer.py`, чтобы собрать `src/` в один markdown-файл (для внешних LLM-инструментов без прямого доступа к файлам) | ОБНОВЛЕНО в тикете 107 |
| `project_packer.py` | Логика pack/unpack | Был синтаксически некорректен (`el` вместо `elif ...` в конце файла) — баг исправлен в тикете 107. С учётом того, что у Claude Code в этом проекте уже есть прямой доступ к файлам, практическая польза инструмента сейчас невелика |
| `backup.bat` | `supabase db dump` с плейсхолдером вместо пароля | Оставлен как есть |

## Резервное копирование БД (Тикет 111)

Отдельный, более новый механизм бэкапа БД — каталог `backup/` в корне
репозитория (`backup/backup-database.ps1`, нативный `pg_dump`/`psql`, ручной
запуск без автоматизации). Не пересекается с `backup.bat`/`project_packer.py`
выше и не описан подробно здесь, так как относится к эксплуатации БД, а не к
сборке/деплою приложения; подробности — в `backup/README.md`.

## Почему

`pack.bat`/`unpack.bat`/`project_packer.py` появились как обходной путь для
AI-инструментов без прямого доступа к файловой системе проекта — отсюда и
пометка тикета 107 о том, что при отсутствии таких внешних инструментов эти
скрипты можно будет убрать.
