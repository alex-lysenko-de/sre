# Обзор архитектуры

Stadtranderholung App — веб-система учёта детей и проведения перекличек во
время поездок и мероприятий (назначение и цели системы — см.
[[Роли-пользователей]] и [[Переклички]] в разделе 02-Предметная-область).
Клиентское SPA-приложение на Vue 3 обращается
напрямую к Supabase (Postgres + Auth + Edge Functions) — без промежуточного
собственного API-слоя (см. [[Целевая-архитектура]] для того, как это должно
измениться).

## Слои приложения

```
Views (страницы, src/views/)
   ↓
Composables (src/composables/useXXX.js) — вся работа с Supabase изолирована здесь
   ↓
Supabase JS SDK → Supabase Cloud (Postgres, Auth, Edge Functions, Realtime)
```

Pinia-store'ы (`stores/user.js`, `stores/config.js`) хранят реактивное
состояние и делегируют операции чтения/записи соответствующим composables —
см. [[stores-user]] как показательный пример разделения
«composable = бизнес-логика/кэш, store = состояние».

## Ключевые технические решения

- **Единственная точка входа к БД** — `src/composables/useXXX.js`,
  минимизирующая связанность с конкретной БД (см. [[Технологический-стек]],
  требование из `readme.md`).
- **Триггеры на стороне БД** (не в клиентском коде) пересчитывают
  агрегированные счётчики присутствия — см. [[Триггеры]].
- **RLS (Row Level Security)** как основной механизм авторизации на уровне
  данных, дополняющий проверки на уровне роутера — см. [[RLS-политики]] и
  [[Карта-маршрутов]].
- **LocalForage** вместо `localStorage` для кэширования пользовательских
  данных — см. [[Кэширование-LocalForage]].
- Отсутствует папка `supabase/migrations/` — источник истины по схеме
  находится в `doc/` и применяется вручную (см. [[Обзор-схемы-БД]]).
