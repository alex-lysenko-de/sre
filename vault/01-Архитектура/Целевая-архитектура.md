# Целевая архитектура

> Источник: `doc/info.md`, раздел «Архитектурные принципы».

## Текущая архитектура

На текущем этапе система использует **Supabase Cloud** как серверную
платформу. Клиент обращается к Supabase напрямую через Supabase JS SDK (см.
[[Обзор-composables]]) — выбрано ради максимально быстрого запуска первой
версии продукта.

## Целевая архитектура

После реализации основной функциональности система должна перейти на
архитектуру, где клиент не зависит от конкретной БД или способа хранения
данных: клиентское приложение работает только через внутренний API
(слой-адаптер). Прямой доступ к БД из клиента — временное решение.

Поддерживаются два режима работы:

- **Cloud Mode** — сервер работает с облачной БД (Supabase или замена).
- **Local Mode** — сервер работает локально на Android-устройстве главного
  администратора.

Переключение между режимами — настройкой адреса API, без изменения
клиентского кода.

## Локальный сервер мероприятия

При полном отсутствии интернета/мобильной сети Android-устройство главного
администратора одновременно:

- включает Wi-Fi Hotspot;
- предоставляет локальный API;
- содержит локальную БД SQLite;
- является единственным сервером мероприятия.

Устройства сопровождающих подключаются к этой точке доступа и продолжают
работать через локальную сеть — без сложной синхронизации между несколькими
устройствами.

## Принцип независимости клиента

Клиент не должен знать, какая БД используется на сервере. Вся работа с
данными — исключительно через API; реализация API может использовать любую
подходящую технологию хранения (Supabase/PostgreSQL или SQLite) без изменения
бизнес-логики и UI.

## Этапы развития проекта

1. Завершить реализацию всей основной функциональности на существующей
   облачной архитектуре.
2. Ввести единый API-слой между клиентом и сервером.
3. Реализовать локальный сервер для Android с SQLite.
4. Добавить переключение между облачным и локальным режимами.
5. Реализовать буферизацию изменений при кратковременных потерях соединения
   с сервером в локальной сети.

## Почему

Прямой доступ к Supabase из клиента ускорил запуск первой версии, но
создаёт зависимость от конкретного backend'а и не работает без интернета
на самом мероприятии (см. [[Работа-без-интернета]] — там уже сейчас
компенсируется локальным кэшированием данных группы, а не полноценным API-
слоем). Целевая архитектура снимает оба ограничения за счёт API-адаптера и
Local Mode.

## Текущий статус (реверс-инжиниринг, Тикет 108)

Ни один из шагов 2–5 пока не реализован: composables обращаются к Supabase
напрямую (`import { supabase } from '@/supabase'`), API-слоя/адаптера в коде
нет. Это подтверждает, что проект всё ещё находится на этапе 1.
