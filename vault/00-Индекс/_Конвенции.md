# Конвенции этого vault (Тикет 108)

Зафиксировано перед началом наполнения — см. `tickets/108/IMPLEMENTATION_PLAN.md`,
раздел «Ключевые решения».

1. **Расположение.** `vault/` — новый каталог в корне репозитория, отдельно от
   `doc/`. `doc/` остаётся как есть (тикет 107) — это «сырой» источник
   (техзадания, SQL, утилиты, `band.psd`), не всё там markdown. Смешивать его
   с vault (принцип «одна тема — один файл» + сквозные wiki-ссылки)
   нецелесообразно.
2. **Плагины Obsidian не нужны.** Используются только базовые возможности
   ядра (wiki-links, backlinks, graph view). Каталог `.obsidian/` в
   репозиторий не коммитится (добавлено в `.gitignore`).
3. **Язык.** Заметки — на русском, как `doc/info.md` и тикеты. Идентификаторы
   из кода (таблицы, поля, composables, маршруты — например
   `presence_morning`, `useArmband.js`, `/admin-busses`) — дословно, без
   перевода. Это не противоречит правилу «no Russian» из `readme.md` — то
   правило про UI/код приложения, а не про внутреннюю документацию.
4. **Атомарность.** Одна заметка = одна сущность/тема (одна таблица БД, один
   composable, одна доменная концепция).
5. **«Почему».** Не отдельные ADR-заметки, а секция «Почему» внутри каждой
   заметки, где это уместно. Исключение — [[Целевая-архитектура]]: переход на
   Cloud/Local Mode и API-адаптер критичен для нескольких разделов сразу,
   поэтому вынесен в отдельную заметку раздела 01, на которую ссылаются
   остальные разделы.
6. **Секреты не переносятся.** `tickets/107/AUDIT_REPORT.md` фиксирует
   реальный `service_role`-ключ в `doc/sb_install_keys.bat` /
   `doc/supabase_install.md` и реальные учётные данные администратора в
   `doc/users.sql`. Заметки о deployment/auth описывают *механизм* (какие
   переменные окружения используются, как называется секрет), но никогда не
   копируют сами значения.

См. [[MOC]] — точка входа в базу знаний.
